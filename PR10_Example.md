# **Кейс: «Автоматизация обработки заявок на участие в образовательных программах»**

---

## 📖 Ситуация

Компания **EduStreamLab** разрабатывает онлайн-образовательные программы и интенсивы для специалистов в сфере IT.

После запуска веб-портала регистрации (разработанного на предыдущей практике) система начала активно использоваться, однако возникли новые задачи:

* необходимо управлять статусами заявок участников;
* удалять ошибочные или дублирующиеся заявки;
* выполнять массовую обработку данных;
* анализировать заявки по заданным критериям.

В связи с этим было принято решение:

* расширить функциональность веб-приложения;
* реализовать более сложный сценарий автоматизации с использованием **Puzzle RPA**.

## 🔘 Детализация кнопок управления

---

### `Mark as Approved`

**Назначение:**
Подтверждение заявки и перевод её в состояние одобренной.

**Бизнес-смысл:**
Заявка прошла проверку и участник допускается к мероприятию.

---

**Когда используется:**

* данные заполнены корректно;
* заявка соответствует требованиям мероприятия;
* есть доступные места (опционально, если реализуется логика ограничения).

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус заявки:

   * если `Status = Pending` → операция разрешена;
   * если `Status ≠ Pending` → операция запрещена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "Approved" }
```

4. В поле `Processed By` записывается текущий пользователь (например, `admin`).
5. Обновляется таблица (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно одобрить уже обработанную заявку;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Approved` или `Rejected`.

---

### `Mark as Rejected`

**Назначение:**
Отклонение заявки.

**Бизнес-смысл:**
Участник не соответствует условиям или заявка содержит ошибки.

---

**Когда используется:**

* некорректные данные;
* превышено количество участников;
* заявка не соответствует требованиям.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. Проверяется статус:

   * допускается только `Pending`.
3. При нажатии:

   * статус изменяется на:

```json
{ "status": "Rejected" }
```

4. В поле `Processed By` записывается пользователь.
5. Таблица обновляется.

---

**Ограничения:**

* повторное отклонение невозможно;
* кнопка `disabled`, если:

  * нет выбранной строки;
  * статус уже обработан (`Approved` / `Rejected`).

---

### `Delete Application`

**Назначение:**
Удаление заявки из системы.

---

**Бизнес-смысл:**

* удаление дубликатов;
* удаление тестовых записей;
* очистка некорректных данных.

---

**Алгоритм работы:**

1. Пользователь выбирает строку.
2. Нажимает кнопку `Delete Application`.

---

**Ограничения:**

* кнопка `disabled`, если:

  * не выбрана строка;
* после удаления:

  * состояние интерфейса должно корректно обновляться.

## 🔍 Фильтрация заявок

Необходимо добавить блок фильтрации:

### `Filter by Status`

**Значения:**

* `All`
* `Pending`
* `Approved`
* `Rejected`

---

### Логика

* при выборе значения отображаются только соответствующие записи
* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by Status`

---

### 📌 Назначение

Фильтр позволяет отображать заявки в зависимости от их текущего состояния (этапа обработки).

---

## 📊 Значения статусов и их смысл

| Статус     | Значение                         | Когда назначается              |
| ---------- | -------------------------------- | ------------------------------ |
| `Pending`  | Заявка создана, но не обработана | при отправке формы             |
| `Approved` | Заявка одобрена                  | при нажатии `Mark as Approved` |
| `Rejected` | Заявка отклонена                 | при нажатии `Mark as Rejected` |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Status
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `Pending`

* отображаются только записи:

```json
{ "status": "Pending" }
```

---

### `Approved`

* отображаются только:

```json
{ "status": "Approved" }
```

---

### `Rejected`

* отображаются только:

```json
{ "status": "Rejected" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No applications found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Pending`, стала `Approved`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

---

#### ✔ Совместно с поиском

Если реализован поиск:

* сначала применяется фильтр по статусу
* затем поиск по имени

---

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий заявки в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* выполнить авторизацию (`login`, `password`)

---

### 2. Загрузка данных

* считать данные из файла
* подготовить коллекцию записей

---

### 3. Добавление заявок

Для каждой записи:

* заполнить форму
* проверить `Submit Application.disabled`
* при необходимости исправить данные
* отправить форму

---

### 4. Анализ заявок

После добавления:

---

#### Найти заявки:

* с `Format = Online`

---

#### Для каждой найденной заявки:

* выбрать строку
* проверить доступность `Mark as Approved`
* изменить статус на:

```json
{ "status": "Approved" }
```

---

### 5. Дополнительная обработка

---

#### Найти заявки:

* с `Format = Offline`

---

#### Для каждой заявки:

* изменить статус на:

```json
{ "status": "Rejected" }
```

---

### 6. Использование фильтра

* выбрать `Filter by Status = Approved`
* убедиться, что отображаются только нужные записи

---

### 7. Поиск

* выполнить поиск по имени (например: `"Ivan"`)
* проверить корректность результата

---

### 8. Удаление записи

* выбрать любую запись

* нажать `Delete Application`

* убедиться, что запись исчезла

---

### 9. Проверка итогового состояния

* проверить:

  * все `Online` → `Approved`
  * все `Offline` → `Rejected`
  * одна запись удалена
