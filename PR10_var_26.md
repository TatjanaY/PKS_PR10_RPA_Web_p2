# **Кейс: «Автоматизация согласования и отклонения заявок на командировки»**

---

## 📖 Ситуация

После запуска базового портала регистрации заявок компания **BusinessTrip Pro** столкнулась с необходимостью управления жизненным циклом заявок на командировки:

* необходимо согласовывать заявки после проверки бюджета;
* отклонять необоснованные или не соответствующие политике заявки;
* удалять неактуальные или ошибочные записи.

В связи с этим было принято решение:

* расширить функциональность веб-приложения;
* реализовать более сложный сценарий автоматизации с использованием **Puzzle RPA**.

## 🔘 Новые кнопки управления

---

### `Approve Trip`

**Назначение:**
Согласование заявки на командировку

**Логика:**

```json
{ "status": "Approved" }
```

**Требования:**

* доступна только если `Status = "Submitted"`
* при нажатии изменяет статус выбранной записи на `"Approved"`

---

### `Reject Trip`

**Назначение:**
Отклонение заявки на командировку

**Логика:**

```json
{ "status": "Rejected" }
```

**Требования:**

* доступна только если `Status = "Submitted"`
* при нажатии изменяет статус выбранной записи на `"Rejected"`

---

### `Delete Request`

**Назначение:**
Удаление заявки из системы

**Логика:**

* удаляет выбранную запись из таблицы

**Требования:**

* доступна только если выбрана строка таблицы
* после удаления таблица обновляется

---

## 🔘 Детализация кнопок управления

---

### `Approve Trip`

**Назначение:**
Согласование заявки после проверки бюджета и обоснования.

**Бизнес-смысл:**
Заявка проверена, командировка согласована для сотрудника.

---

**Когда используется:**

* руководитель проверил обоснование поездки;
* бюджет позволяет оплатить командировку.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус заявки:

   * если `Status ≠ Submitted` → операция запрещена;
   * если `Status = Submitted` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "Approved" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно согласовать уже обработанную заявку;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Approved` или `Rejected`.

---

### `Reject Trip`

**Назначение:**
Отклонение заявки на командировку.

**Бизнес-смысл:**
Заявка не соответствует политике или бюджетным ограничениям.

---

**Когда используется:**

* недостаточное обоснование поездки;
* превышен бюджет подразделения.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. Проверяется статус:

   * допускается только `Submitted`.
3. При нажатии:

   * статус изменяется на:

```json
{ "status": "Rejected" }
```

4. Таблица обновляется.

---

**Ограничения:**

* повторное отклонение невозможно;
* кнопка `disabled`, если:

  * нет выбранной строки;
  * статус уже обработан.

---

### `Delete Request`

**Назначение:**
Удаление заявки из системы.

**Бизнес-смысл:**
Заявка более не актуальна, запись удаляется.

---

**Когда используется:**

* сотрудник отменил запрос;
* запись была создана ошибочно.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. При нажатии:

   * запись удаляется из таблицы;
3. Таблица обновляется.

---

**Ограничения:**

* кнопка `disabled`, если:

  * нет выбранной строки;
* после удаления интерфейс корректно обновляется.

## 🔍 Фильтрация заявок

Необходимо добавить блок фильтрации:

### `Filter by Purpose`

**Значения:**

* `All`
* `Conference`
* `Business Meeting`
* `Training`

---

### Логика

* при выборе значения отображаются только соответствующие записи
* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by Purpose`

---

### 📌 Назначение

Фильтр позволяет отображать заявки в зависимости от цели командировки.

---

## 📊 Значения целей и их смысл

| Цель | Значение | Когда используется |
| ---- | -------- | ------------------ |
| `Conference` | Участие в конференции | заявки на профессиональные события |
| `Business Meeting` | Деловая встреча | заявки на переговоры с партнёрами |
| `Training` | Обучение | заявки на повышение квалификации |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Purpose
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `Conference`

* отображаются только записи:

```json
{ "purpose": "Conference" }
```

---

### `Business Meeting`

* отображаются только:

```json
{ "purpose": "Business Meeting" }
```

---

### `Training`

* отображаются только:

```json
{ "purpose": "Training" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No trip requests found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Submitted`, стала `Approved`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий заявки в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление заявок

Для каждой записи:

* заполнить форму
* проверить `Submit Request.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Одобрение заявок с целью Conference

* найти заявки с `Purpose = Conference`

* для каждой найденной заявки:

  * выбрать строку
  * проверить доступность `Approve Trip`
  * изменить статус на:

```json
{ "status": "Approved" }
```

---

### 4. Использование фильтра

* выбрать `Filter by Purpose = Conference`
* убедиться, что отображаются только нужные записи

---

### 5. Поиск

* выполнить поиск по имени сотрудника (например: `"Ivan"`)
* проверить корректность результата

---

### 6. Отклонение заявки

* выбрать любую запись со статусом `Submitted`
* нажать `Reject Trip`
* убедиться, что статус изменился на `"Rejected"`

---

### 7. Удаление записи

* выбрать любую запись
* нажать `Delete Request`
* убедиться, что запись исчезла

---

### 8. Проверка итогового состояния

* проверить:

  * все заявки добавлены (10 записей)
  * заявки с `Purpose = Conference` имеют статус `"Approved"`
  * одна заявка отклонена
  * одна запись удалена
