* расширить функциональность веб-приложения;

## 🔘 Новые кнопки управления

---

### `Approve Visit`

**Назначение:**
Подтверждение визита посетителя

**Логика:**

```json
{ "status": "Approved" }
```

**Требования:**

* доступна только если `Status = "Planned"`
* при нажатии изменяет статус выбранной записи на `"Approved"`

---

### `Cancel Visit`

**Назначение:**
Отмена визита посетителя

**Логика:**

```json
{ "status": "Cancelled" }
```

**Требования:**

* доступна только если `Status = "Planned"`
* при нажатии изменяет статус выбранной записи на `"Cancelled"`

## 🔘 Детализация кнопок управления

---

### `Approve Visit`

**Назначение:**
Подтверждение визита после проверки расписания.

**Бизнес-смысл:**
Визит согласован, пропуск оформлен для посетителя.

---

**Когда используется:**

* администратор проверил доступность принимающего сотрудника;
* визит соответствует правилам безопасности.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус заявки:

   * если `Status ≠ Planned` → операция запрещена;
   * если `Status = Planned` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "Approved" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно подтвердить уже обработанный визит;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Approved` или `Cancelled`.

---

### `Cancel Visit`

**Назначение:**
Отмена визита посетителя.

**Бизнес-смысл:**
Визит более не актуален, пропуск аннулируется.

---

**Когда используется:**

* посетитель отменил встречу;
* визит был зарегистрирован ошибочно.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. Проверяется статус:

   * допускается только `Planned`.
3. При нажатии:

   * статус изменяется на:

```json
{ "status": "Cancelled" }
```

4. Таблица обновляется.

---

**Ограничения:**

* повторная отмена невозможна;
* кнопка `disabled`, если:

  * нет выбранной строки;
  * статус уже обработан.

## 🔍 Фильтрация визитов

Необходимо добавить блок фильтрации:

### `Filter by Purpose`

**Значения:**

* `All`
* `Meeting`
* `Interview`
* `Delivery`
* `Other`

---

### Логика

* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by Purpose`

---

### 📌 Назначение

Фильтр позволяет отображать визиты в зависимости от цели посещения.

---

## 📊 Значения целей визита и их смысл

| Цель | Значение | Когда используется |
| ---- | -------- | ------------------ |
| `Meeting` | Деловая встреча | визиты клиентов и партнёров |
| `Interview` | Собеседование | визиты кандидатов на работу |
| `Delivery` | Доставка | визиты курьеров и поставщиков |
| `Other` | Прочее | иные цели посещения |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Purpose
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `Meeting`

* отображаются только записи:

```json
{ "purpose": "Meeting" }
```

---

### `Interview`

* отображаются только:

```json
{ "purpose": "Interview" }
```

---

### `Delivery`

* отображаются только:

```json
{ "purpose": "Delivery" }
```

---

### `Other`

* отображаются только:

```json
{ "purpose": "Other" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No visits found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Planned`, стала `Approved`)

---

#### ✔ После отмены

* таблица автоматически обновляется с учётом текущего фильтра

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий визиты в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление визитов

Для каждой записи:

* заполнить форму
* проверить `Register Visit.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Подтверждение визитов типа Meeting

* найти визиты с `Purpose = Meeting`

* для каждой найденной записи:

  * выбрать строку
  * проверить доступность `Approve Visit`
  * изменить статус на:

```json
{ "status": "Approved" }
```

---

### 4. Использование фильтра

* выбрать `Filter by Purpose = Meeting`
* убедиться, что отображаются только нужные записи

---

### 5. Поиск

* выполнить поиск по имени посетителя (например: `"Alexei"`)
* проверить корректность результата

---

### 6. Отмена визита

* выбрать любую запись со статусом `Planned`
* нажать `Cancel Visit`
* убедиться, что статус изменился на `"Cancelled"`

---

### 7. Проверка итогового состояния

* проверить:

  * все визиты добавлены (10 записей)
  * визиты с `Purpose = Meeting` имеют статус `"Approved"`
  * одна запись отменена
