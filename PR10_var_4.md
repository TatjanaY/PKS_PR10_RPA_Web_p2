# **Кейс: «Автоматизация подтверждения и управления заказами на офисное оборудование»**

---

## 📖 Ситуация

После запуска базового портала регистрации заявок компания **OfficeSupply Pro** столкнулась с необходимостью управления жизненным циклом заказов:

* необходимо подтверждать заказы после проверки наличия оборудования;
* отменять неактуальные или ошибочные заявки;
* отслеживать статус выполнения заказов.

В связи с этим было принято решение:

* расширить функциональность веб-приложения;
* реализовать более сложный сценарий автоматизации с использованием **Puzzle RPA**.

# **Задание 1. Расширение функциональности веб-приложения**

Необходимо доработать существующее веб-приложение, добавив новые элементы управления и логику обработки заказов.

**Каждый элемент интерфейса должен иметь уникальный `id`**

---

## 🔘 Новые кнопки управления

---

### `Approve Order`

**Назначение:**
Подтверждение заказа на оборудование

**Логика:**

```json
{ "status": "Approved" }
```

**Требования:**

* доступна только если `Status ≠ "Approved"`
* при нажатии изменяет статус выбранной записи на `"Approved"`

---

### `Delete Order`

**Назначение:**
Удаление заказа из системы

**Логика:**

* удаляет выбранную запись из таблицы

**Требования:**

* доступна только если выбрана строка таблицы
* после удаления таблица обновляется

---

## ⚙️ Управление состоянием элементов (`disabled`)

---

### Кнопки `Approve Order`, `Delete Order`

Должны быть:

* `disabled = true`, если:

  * ни одна строка не выбрана

* `disabled = false`, если:

  * выбрана строка таблицы

---

### Дополнительные условия

* `Approve Order` должна быть неактивна, если:

```json
{ "status": "Approved" }
```

## 🔘 Детализация кнопок управления

---

### `Approve Order`

**Назначение:**
Подтверждение заказа после проверки наличия оборудования.

**Бизнес-смысл:**
Заказ проверен, оборудование зарезервировано для сотрудника.

---

**Когда используется:**

* администратор проверил наличие на складе;
* заказ соответствует политике компании.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус заявки:

   * если `Status = Approved` → операция запрещена;
   * если `Status ≠ Approved` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "Approved" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно подтвердить уже обработанный заказ;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Approved`.

---

### `Delete Order`

**Назначение:**
Удаление заказа из системы.

**Бизнес-смысл:**
Заявка более не актуальна, заказ отменяется.

---

**Когда используется:**

* сотрудник отменил запрос;
* заказ был создан ошибочно.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. При нажатии:

   * запись удаляется из таблицы;
3. Таблица обновляется.

---

**Ограничения:**

* кнопка `disabled`, если:

  * нет выбранной строки;
* после удаления интерфейс корректно обновляется.

## 🔍 Фильтрация заказов

Необходимо добавить блок фильтрации:

### `Filter by Department`

**Значения:**

* `All`
* `IT`
* `HR`
* `Finance`

---

### Логика

* при выборе значения отображаются только соответствующие записи
* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by Department`

---

### 📌 Назначение

Фильтр позволяет отображать заказы в зависимости от подразделения сотрудника.

---

## 📊 Значения подразделений и их смысл

| Подразделение | Значение | Когда используется |
| ------------- | -------- | ------------------ |
| `IT` | Отдел информационных технологий | заявки от IT-специалистов |
| `HR` | Отдел кадров | заявки от HR-менеджеров |
| `Finance` | Финансовый отдел | заявки от бухгалтерии |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Department
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `IT`

* отображаются только записи:

```json
{ "department": "IT" }
```

---

### `HR`

* отображаются только:

```json
{ "department": "HR" }
```

---

### `Finance`

* отображаются только:

```json
{ "department": "Finance" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No orders found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `New`, стала `Approved`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

---

## 🎯 Итоговая логика состояний

```text
Создание заказа → New
             ↓
          Approved
```

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий заказы в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление заказов

Для каждой записи:

* заполнить форму
* проверить `Create Order.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Подтверждение заказов

После добавления:

* для каждой записи в таблице:

  * выбрать строку
  * проверить доступность `Approve Order`
  * изменить статус на:

```json
{ "status": "Approved" }
```

---

### 4. Использование фильтра

* выбрать `Filter by Department = IT`
* убедиться, что отображаются только нужные записи

---

### 5. Поиск

* выполнить поиск по наименованию (например: `"Laptop"`)
* проверить корректность результата

---

### 6. Удаление записи

* выбрать любую запись
* нажать `Delete Order`
* убедиться, что запись исчезла

---

### 7. Проверка итогового состояния

* проверить:

  * все заказы добавлены (10 записей)
  * половина заказов имеет статус `"Approved"`
  * одна запись удалена
