# **Кейс: «Автоматизация сервисного обслуживания лабораторного оборудования»**

---

## 📖 Ситуация

После запуска базового портала регистрации оборудования компания **LabEquipment Pro** столкнулась с необходимостью управления жизненным циклом сервисного обслуживания:

* необходимо отправлять оборудование в сервис при выявлении неисправностей;
* отмечать завершение ремонта и возврат оборудования в эксплуатацию;
* удалять списанное или неактуальное оборудование.

В связи с этим было принято решение:

* расширить функциональность веб-приложения;
* реализовать более сложный сценарий автоматизации с использованием **Puzzle RPA**.

## 🔘 Новые кнопки управления

---

### `Send to Service`

**Назначение:**
Отправка оборудования в сервисный центр

**Логика:**

```json
{ "status": "In Service" }
```

**Требования:**

* доступна только если `Condition = "Needs Service"`
* при нажатии изменяет статус выбранной записи на `"In Service"`

---

### `Mark as Repaired`

**Назначение:**
Отметка оборудования как отремонтированного

**Логика:**

```json
{ "status": "Repaired" }
```

**Требования:**

* доступна только если `Status = "In Service"`
* при нажатии изменяет статус выбранной записи на `"Repaired"`

---

### `Delete Equipment`

**Назначение:**
Удаление оборудования из системы

**Логика:**

* удаляет выбранную запись из таблицы

**Требования:**

* доступна только если выбрана строка таблицы
* после удаления таблица обновляется

---

## ⚙️ Управление состоянием элементов (`disabled`)

---

### Кнопки `Send to Service`, `Mark as Repaired`, `Delete Equipment`

Должны быть:

* `disabled = true`, если:

  * ни одна строка не выбрана

* `disabled = false`, если:

  * выбрана строка таблицы

---

### Дополнительные условия

* `Send to Service` должна быть неактивна, если:

```json
{ "condition": "Good" }
или
{ "condition": "Broken" }
```

* `Mark as Repaired` должна быть неактивна, если:

```json
{ "status": "Repaired" }
```

## 🔘 Детализация кнопок управления

---

### `Send to Service`

**Назначение:**
Отправка оборудования в сервисный центр после выявления неисправности.

**Бизнес-смысл:**
Оборудование требует ремонта, передано в сервис.

---

**Когда используется:**

* проведена диагностика оборудования;
* выявлена необходимость сервисного обслуживания.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущее состояние оборудования:

   * если `Condition ≠ Needs Service` → операция запрещена;
   * если `Condition = Needs Service` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "In Service" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя отправить в сервис оборудование, не требующее ремонта;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * условие `Condition = Needs Service` не выполнено.

---

### `Mark as Repaired`

**Назначение:**
Отметка оборудования как отремонтированного и готового к эксплуатации.

**Бизнес-смысл:**
Ремонт завершён, оборудование возвращено в лабораторию.

---

**Когда используется:**

* сервисный центр завершил ремонт;
* проведено тестирование оборудования.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. Проверяется статус:

   * допускается только `In Service`.
3. При нажатии:

   * статус изменяется на:

```json
{ "status": "Repaired" }
```

4. Таблица обновляется.

---

**Ограничения:**

* нельзя отметить как отремонтированное оборудование, не находящееся в сервисе;
* кнопка `disabled`, если:

  * нет выбранной строки;
  * статус уже `Repaired`.

---

### `Delete Equipment`

**Назначение:**
Удаление оборудования из системы.

**Бизнес-смысл:**
Оборудование списано или более не учитывается.

---

**Когда используется:**

* оборудование списано по акту;
* запись была создана ошибочно.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. При нажатии:

   * запись удаляется из таблицы;
3. Таблица обновляется.

---

**Ограничения:**

* кнопка `disabled`, если:

  * нет выбранной строки;
* после удаления интерфейс корректно обновляется.

## 🔍 Детализация фильтрации `Filter by Lab`

---

### 📌 Назначение

Фильтр позволяет отображать оборудование в зависимости от лаборатории.

---

## 📊 Значения лабораторий и их смысл

| Лаборатория | Значение | Когда используется |
| ----------- | -------- | ------------------ |
| `AI Lab` | Лаборатория искусственного интеллекта | оборудование для ML-исследований |
| `Robotics Lab` | Робототехническая лаборатория | оборудование для робототехники |
| `Data Lab` | Лаборатория данных | оборудование для обработки данных |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Lab
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `AI Lab`

* отображаются только записи:

```json
{ "lab": "AI Lab" }
```

---

### `Robotics Lab`

* отображаются только:

```json
{ "lab": "Robotics Lab" }
```

---

### `Data Lab`

* отображаются только:

```json
{ "lab": "Data Lab" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No equipment found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Registered`, стала `Repaired`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий оборудование в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление оборудования

Для каждой записи:

* заполнить форму
* проверить `Add Equipment.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Отправка в сервис оборудования, требующего обслуживания

* найти оборудование с `Condition = Needs Service`

* для каждой найденной записи:

  * выбрать строку
  * проверить доступность `Send to Service`
  * изменить статус на:

```json
{ "status": "In Service" }
```

---

### 4. Отметка отремонтированного оборудования

* для каждой записи со статусом `In Service`:

  * выбрать строку
  * проверить доступность `Mark as Repaired`
  * изменить статус на:

```json
{ "status": "Repaired" }
```

---

### 5. Использование фильтра

* выбрать `Filter by Lab = AI Lab`
* убедиться, что отображаются только нужные записи

---

### 6. Поиск

* выполнить поиск по названию оборудования (например: `"Microscope"`)
* проверить корректность результата

---

### 7. Удаление записи

* выбрать любую запись
* нажать `Delete Equipment`
* убедиться, что запись исчезла

---

### 8. Проверка итогового состояния

* проверить:

  * всё оборудование добавлено (10 записей)
  * оборудование с `Condition = Needs Service` имеет статус `"Repaired"`
  * одна запись удалена
