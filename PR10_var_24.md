## 📖 Ситуация

После запуска базового портала регистрации заявок компания **AccessControl Pro** столкнулась с необходимостью управления жизненным циклом запросов на доступ:

* необходимо согласовывать заявки после проверки полномочий;
* отклонять необоснованные или не соответствующие политике заявки;
* удалять неактуальные или ошибочные записи.

В связи с этим было принято решение:

* расширить функциональность веб-приложения;
* реализовать более сложный сценарий автоматизации с использованием **Puzzle RPA**.

## 🔘 Новые кнопки управления

---

### `Approve Access`

**Назначение:**
Согласование заявки на доступ

**Логика:**

```json
{ "status": "Approved" }
```

**Требования:**

* доступна только если `Status = "Pending"`
* при нажатии изменяет статус выбранной записи на `"Approved"`

---

### `Deny Access`

**Назначение:**
Отклонение заявки на доступ

**Логика:**

```json
{ "status": "Denied" }
```

**Требования:**

* доступна только если `Status = "Pending"`
* при нажатии изменяет статус выбранной записи на `"Denied"`

---

### `Delete Request`

**Назначение:**
Удаление заявки из системы

**Логика:**

* удаляет выбранную запись из таблицы

**Требования:**

* доступна только если выбрана строка таблицы
* после удаления таблица обновляется

---

## ⚙️ Управление состоянием элементов (`disabled`)

---

### Кнопки `Approve Access`, `Deny Access`, `Delete Request`

Должны быть:

* `disabled = true`, если:

  * ни одна строка не выбрана

* `disabled = false`, если:

  * выбрана строка таблицы

---

### Дополнительные условия

* `Approve Access` должна быть неактивна, если:

```json
{ "status": "Approved" }
или
{ "status": "Denied" }
```

* `Deny Access` должна быть неактивна, если:

```json
{ "status": "Approved" }
или
{ "status": "Denied" }
```

## 🔘 Детализация кнопок управления

---

### `Approve Access`

**Назначение:**
Согласование заявки после проверки полномочий сотрудника.

**Бизнес-смысл:**
Заявка проверена, доступ предоставлен сотруднику.

---

**Когда используется:**

* администратор проверил обоснование доступа;
* сотрудник имеет соответствующие полномочия.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус заявки:

   * если `Status ≠ Pending` → операция запрещена;
   * если `Status = Pending` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "Approved" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно согласовать уже обработанную заявку;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Approved` или `Denied`.

---

### `Deny Access`

**Назначение:**
Отклонение заявки на доступ.

**Бизнес-смысл:**
Заявка не соответствует политике безопасности или не обоснована.

---

**Когда используется:**

* недостаточное обоснование доступа;
* сотрудник не имеет полномочий для запрашиваемого уровня.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. Проверяется статус:

   * допускается только `Pending`.
3. При нажатии:

   * статус изменяется на:

```json
{ "status": "Denied" }
```

4. Таблица обновляется.

---

**Ограничения:**

* повторное отклонение невозможно;
* кнопка `disabled`, если:

  * нет выбранной строки;
  * статус уже обработан.

---

### `Delete Request`

**Назначение:**
Удаление заявки из системы.

**Бизнес-смысл:**
Заявка более не актуальна, запись удаляется.

---

**Когда используется:**

* сотрудник отменил запрос;
* запись была создана ошибочно.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. При нажатии:

   * запись удаляется из таблицы;
3. Таблица обновляется.

---

**Ограничения:**

* кнопка `disabled`, если:

  * нет выбранной строки;
* после удаления интерфейс корректно обновляется.

## 🔍 Фильтрация заявок

Необходимо добавить блок фильтрации:

### `Filter by System`

**Значения:**

* `All`
* `CRM`
* `ERP`
* `Analytics`

---

### Логика

* при выборе значения отображаются только соответствующие записи
* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by System`

---

### 📌 Назначение

Фильтр позволяет отображать заявки в зависимости от запрашиваемой системы.

---

## 📊 Значения систем и их смысл

| Система | Значение | Когда используется |
| ------- | -------- | ------------------ |
| `CRM` | Система управления клиентами | заявки на доступ к клиентской базе |
| `ERP` | Система планирования ресурсов | заявки на доступ к финансовым данным |
| `Analytics` | Аналитическая платформа | заявки на доступ к отчётам |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by System
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `CRM`

* отображаются только записи:

```json
{ "system": "CRM" }
```

---

### `ERP`

* отображаются только:

```json
{ "system": "ERP" }
```

---

### `Analytics`

* отображаются только:

```json
{ "system": "Analytics" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No access requests found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Pending`, стала `Approved`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий заявки в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление заявок

Для каждой записи:

* заполнить форму
* проверить `Submit Request.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Одобрение заявок на систему CRM

* найти заявки с `System = CRM`

* для каждой найденной заявки:

  * выбрать строку
  * проверить доступность `Approve Access`
  * изменить статус на:

```json
{ "status": "Approved" }
```

---

### 4. Использование фильтра

* выбрать `Filter by System = CRM`
* убедиться, что отображаются только нужные записи

---

### 5. Поиск

* выполнить поиск по имени сотрудника (например: `"Ivan"`)
* проверить корректность результата

---

### 6. Отклонение заявки

* выбрать любую запись со статусом `Pending`
* нажать `Deny Access`
* убедиться, что статус изменился на `"Denied"`

---

### 7. Удаление записи

* выбрать любую запись
* нажать `Delete Request`
* убедиться, что запись исчезла

---

### 8. Проверка итогового состояния

* проверить:

  * все заявки добавлены (10 записей)
  * заявки на систему `CRM` имеют статус `"Approved"`
  * одна заявка отклонена
  * одна запись удалена
