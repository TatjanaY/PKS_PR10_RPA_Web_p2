## 📖 Ситуация

После запуска базового портала регистрации обращений компания **ServiceCenter Pro** столкнулась с необходимостью управления жизненным циклом заявок на ремонт:

* необходимо начинать ремонт после классификации проблемы;
* завершать ремонт после устранения неисправности;
* удалять неактуальные или ошибочные записи.

В связи с этим было принято решение:

* расширить функциональность веб-приложения;
* реализовать более сложный сценарий автоматизации с использованием **Puzzle RPA**.

## 🔘 Новые кнопки управления

---

### `Start Repair`

**Назначение:**
Начало ремонта устройства

**Логика:**

```json
{ "status": "In Repair" }
```

**Требования:**

* доступна только если `Status ≠ "Completed"`
* при нажатии изменяет статус выбранной записи на `"In Repair"`

---

### `Complete Repair`

**Назначение:**
Завершение ремонта устройства

**Логика:**

```json
{ "status": "Completed" }
```

**Требования:**

* доступна только если `Status = "In Repair"`
* при нажатии изменяет статус выбранной записи на `"Completed"`

---

### `Delete Request`

**Назначение:**
Удаление обращения из системы

**Логика:**

* удаляет выбранную запись из таблицы

**Требования:**

* доступна только если выбрана строка таблицы
* после удаления таблица обновляется

---

## ⚙️ Управление состоянием элементов (`disabled`)

---

### Кнопки `Start Repair`, `Complete Repair`, `Delete Request`

Должны быть:

* `disabled = true`, если:

  * ни одна строка не выбрана

* `disabled = false`, если:

  * выбрана строка таблицы

---

### Дополнительные условия

* `Complete Repair` должна быть неактивна, если:

```json
{ "status": "Completed" }
```

## 🔘 Детализация кнопок управления

---

### `Start Repair`

**Назначение:**
Начало ремонта после классификации проблемы.

**Бизнес-смысл:**
Мастер приступил к устранению неисправности устройства.

---

**Когда используется:**

* обращение прошло первичную диагностику;
* определён тип проблемы и приоритет.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус обращения:

   * если `Status = Completed` → операция запрещена;
   * если `Status ≠ Completed` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "In Repair" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно начать уже ремонтируемое обращение;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Completed`.

---

### `Complete Repair`

**Назначение:**
Завершение ремонта устройства.

**Бизнес-смысл:**
Неисправность устранена, устройство готово к выдаче.

---

**Когда используется:**

* мастер устранил проблему;
* проведено тестирование устройства.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. Проверяется статус:

   * допускается только `In Repair`.
3. При нажатии:

   * статус изменяется на:

```json
{ "status": "Completed" }
```

4. Таблица обновляется.

---

**Ограничения:**

* нельзя завершить ремонт, который не был начат;
* кнопка `disabled`, если:

  * нет выбранной строки;
  * статус уже `Completed`.

---

### `Delete Request`

**Назначение:**
Удаление обращения из системы.

**Бизнес-смысл:**
Обращение более не актуально, запись удаляется.

---

**Когда используется:**

* клиент отменил заявку;
* запись была создана ошибочно.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. При нажатии:

   * запись удаляется из таблицы;
3. Таблица обновляется.

---

**Ограничения:**

* кнопка `disabled`, если:

  * нет выбранной строки;
* после удаления интерфейс корректно обновляется.

## 🔍 Фильтрация обращений

Необходимо добавить блок фильтрации:

### `Filter by Problem Type`

**Значения:**

* `All`
* `Hardware`
* `Software`
* `Network`

---

### Логика

* при выборе значения отображаются только соответствующие записи
* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by Problem Type`

---

### 📌 Назначение

Фильтр позволяет отображать обращения в зависимости от типа проблемы.

---

## 📊 Значения типов проблем и их смысл

| Тип проблемы | Значение | Когда используется |
| ------------ | -------- | ------------------ |
| `Hardware` | Аппаратная неисправность | поломка компонентов устройства |
| `Software` | Программная ошибка | сбои в работе ПО |
| `Network` | Сетевая проблема | проблемы с подключением |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Problem Type
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `Hardware`

* отображаются только записи:

```json
{ "problem_type": "Hardware" }
```

---

### `Software`

* отображаются только:

```json
{ "problem_type": "Software" }
```

---

### `Network`

* отображаются только:

```json
{ "problem_type": "Network" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No service requests found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Registered`, стала `Completed`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий обращения в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление обращений

Для каждой записи:

* заполнить форму
* проверить `Register Request.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Начало ремонта обращений с высоким приоритетом

* найти обращения с `Priority = High`

* для каждой найденной записи:

  * выбрать строку
  * проверить доступность `Start Repair`
  * изменить статус на:

```json
{ "status": "In Repair" }
```

---

### 4. Завершение ремонта обращений с высоким приоритетом

* для каждой записи со статусом `In Repair` и `Priority = High`:

  * выбрать строку
  * проверить доступность `Complete Repair`
  * изменить статус на:

```json
{ "status": "Completed" }
```

---

### 5. Использование фильтра

* выбрать `Filter by Problem Type = Hardware`
* убедиться, что отображаются только нужные записи

---

### 6. Поиск

* выполнить поиск по имени клиента (например: `"Ivan"`)
* проверить корректность результата

---

### 7. Проверка итогового состояния

* проверить:

  * все обращения добавлены (10 записей)
  * обращения с `Priority = High` имеют статус `"Completed"`
