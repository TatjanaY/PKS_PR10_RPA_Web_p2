# **Кейс: «Автоматизация процесса ремонта оборудования»**

---

## 📖 Ситуация

После запуска базового портала регистрации заявок компания **ServiceTech Pro** столкнулась с необходимостью управления жизненным циклом заявок на ремонт:

* необходимо начинать ремонт после проверки оборудования;
* завершать ремонт после устранения неисправности;
* отслеживать статус выполнения работ.

В связи с этим было принято решение:

* расширить функциональность веб-приложения;
* реализовать более сложный сценарий автоматизации с использованием **Puzzle RPA**.

---

# **Задание 1. Расширение функциональности веб-приложения**

Необходимо доработать существующее веб-приложение, добавив новые элементы управления и логику обработки заявок.

**Каждый элемент интерфейса должен иметь уникальный `id`**

---

## 🔘 Новые кнопки управления

---

### `Start Repair`

**Назначение:**
Начало ремонта оборудования

**Логика:**

```json
{ "status": "In Progress" }
```

**Требования:**

* доступна только если `Status = "Open"`
* при нажатии изменяет статус выбранной записи на `"In Progress"`

---

### `Complete Repair`

**Назначение:**
Завершение ремонта оборудования

**Логика:**

```json
{ "status": "Completed" }
```

**Требования:**

* доступна только если `Status = "In Progress"`
* при нажатии изменяет статус выбранной записи на `"Completed"`

---

## ⚙️ Управление состоянием элементов (`disabled`)

---

### Кнопки `Start Repair`, `Complete Repair`

Должны быть:

* `disabled = true`, если:

  * ни одна строка не выбрана

* `disabled = false`, если:

  * выбрана строка таблицы

---

### Дополнительные условия

* `Start Repair` должна быть неактивна, если:

```json
{ "status": "In Progress" }
или
{ "status": "Completed" }
```

* `Complete Repair` должна быть неактивна, если:

```json
{ "status": "Open" }
или
{ "status": "Completed" }
```

## 🔘 Детализация кнопок управления

---

### `Start Repair`

**Назначение:**
Начало ремонта после проверки оборудования.

**Бизнес-смысл:**
Техник приступил к устранению неисправности.

---

**Когда используется:**

* техник проверил оборудование;
* определена причина поломки.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус заявки:

   * если `Status ≠ Open` → операция запрещена;
   * если `Status = Open` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "In Progress" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно начать уже обрабатываемую заявку;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `In Progress` или `Completed`.

---

### `Complete Repair`

**Назначение:**
Завершение ремонта оборудования.

**Бизнес-смысл:**
Неисправность устранена, оборудование готово к эксплуатации.

---

**Когда используется:**

* техник устранил неисправность;
* проведено тестирование оборудования.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. Проверяется статус:

   * допускается только `In Progress`.
3. При нажатии:

   * статус изменяется на:

```json
{ "status": "Completed" }
```

4. Таблица обновляется.

---

**Ограничения:**

* нельзя завершить ремонт, который не был начат;
* кнопка `disabled`, если:

  * нет выбранной строки;
  * статус уже обработан.

## 🔍 Фильтрация заявок

Необходимо добавить блок фильтрации:

### `Filter by Priority`

**Значения:**

* `All`
* `Low`
* `Medium`
* `High`

---

### Логика

* при выборе значения отображаются только соответствующие записи
* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by Priority`

---

### 📌 Назначение

Фильтр позволяет отображать заявки в зависимости от приоритета ремонта.

---

## 📊 Значения приоритетов и их смысл

| Приоритет | Значение | Когда назначается |
| --------- | -------- | ----------------- |
| `Low` | Низкий приоритет | при незначительных неисправностях |
| `Medium` | Средний приоритет | при стандартных поломках |
| `High` | Высокий приоритет | при критических сбоях |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Priority
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `Low`

* отображаются только записи:

```json
{ "priority": "Low" }
```

---

### `Medium`

* отображаются только:

```json
{ "priority": "Medium" }
```

---

### `High`

* отображаются только:

```json
{ "priority": "High" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No repair requests found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Open`, стала `Completed`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

---

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий заявки в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление заявок

Для каждой записи:

* заполнить форму
* проверить `Create Repair Request.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Начало ремонта всех заявок

После добавления:

* для каждой записи в таблице:

  * выбрать строку
  * проверить доступность `Start Repair`
  * изменить статус на:

```json
{ "status": "In Progress" }
```

---

### 4. Завершение ремонта заявок с высоким приоритетом

* найти заявки с `Priority = High`

* для каждой найденной заявки:

  * выбрать строку
  * проверить доступность `Complete Repair`
  * изменить статус на:

```json
{ "status": "Completed" }
```

---

### 5. Использование фильтра

* выбрать `Filter by Priority = High`
* убедиться, что отображаются только нужные записи

---

### 6. Поиск

* выполнить поиск по наименованию оборудования (например: `"Printer"`)
* проверить корректность результата

---

### 7. Проверка итогового состояния

* проверить:

  * все заявки добавлены (10 записей)
  * все записи имеют статус `"In Progress"` или `"Completed"`
  * заявки с `Priority = High` имеют статус `"Completed"`
