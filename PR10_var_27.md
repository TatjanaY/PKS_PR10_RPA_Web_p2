# **Кейс: «Автоматизация согласования и доставки заказов на канцелярию»**

---

## 📖 Ситуация

После запуска базового портала регистрации заказов компания **OfficeSupplies Pro** столкнулась с необходимостью управления жизненным циклом заявок:

* необходимо согласовывать заказы после проверки наличия товаров;
* отмечать доставку товаров сотрудникам;
* удалять неактуальные или ошибочные записи.

В связи с этим было принято решение:

* расширить функциональность веб-приложения;
* реализовать более сложный сценарий автоматизации с использованием **Puzzle RPA**.

# **Задание 1. Расширение функциональности веб-приложения**

Необходимо доработать существующее веб-приложение, добавив новые элементы управления и логику обработки заказов.

**Каждый элемент интерфейса должен иметь уникальный `id`**

---

## 🔘 Новые кнопки управления

---

### `Approve Order`

**Назначение:**
Согласование заказа на канцелярию

**Логика:**

```json
{ "status": "Approved" }
```

**Требования:**

* доступна только если `Status = "New"`
* при нажатии изменяет статус выбранной записи на `"Approved"`

---

### `Deliver Order`

**Назначение:**
Отметка о доставке заказа

**Логика:**

```json
{ "status": "Delivered" }
```

**Требования:**

* доступна только если `Status = "Approved"`
* при нажатии изменяет статус выбранной записи на `"Delivered"`

---

### `Delete Order`

**Назначение:**
Удаление заказа из системы

**Логика:**

* удаляет выбранную запись из таблицы

**Требования:**

* доступна только если выбрана строка таблицы
* после удаления таблица обновляется

---

## ⚙️ Управление состоянием элементов (`disabled`)

---

### Кнопки `Approve Order`, `Deliver Order`, `Delete Order`

Должны быть:

* `disabled = true`, если:

  * ни одна строка не выбрана

* `disabled = false`, если:

  * выбрана строка таблицы

---

### Дополнительные условия

* `Approve Order` должна быть неактивна, если:

```json
{ "status": "Approved" }
или
{ "status": "Delivered" }
```

* `Deliver Order` должна быть неактивна, если:

```json
{ "status": "Delivered" }
```

## 🔘 Детализация кнопок управления

---

### `Approve Order`

**Назначение:**
Согласование заказа после проверки наличия товаров.

**Бизнес-смысл:**
Заказ проверен, товары зарезервированы для сотрудника.

---

**Когда используется:**

* администратор проверил наличие на складе;
* заказ соответствует политике компании.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус заказа:

   * если `Status ≠ New` → операция запрещена;
   * если `Status = New` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "Approved" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно согласовать уже обработанный заказ;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Approved` или `Delivered`.

---

### `Deliver Order`

**Назначение:**
Отметка о доставке заказа сотруднику.

**Бизнес-смысл:**
Товары переданы сотруднику, заказ выполнен.

---

**Когда используется:**

* товары подготовлены к выдаче;
* сотрудник получил заказ.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. Проверяется статус:

   * допускается только `Approved`.
3. При нажатии:

   * статус изменяется на:

```json
{ "status": "Delivered" }
```

4. Таблица обновляется.

---

**Ограничения:**

* нельзя доставить заказ, который не был согласован;
* кнопка `disabled`, если:

  * нет выбранной строки;
  * статус уже `Delivered`.

---

### `Delete Order`

**Назначение:**
Удаление заказа из системы.

**Бизнес-смысл:**
Заказ более не актуален, запись удаляется.

---

**Когда используется:**

* сотрудник отменил заказ;
* запись была создана ошибочно.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. При нажатии:

   * запись удаляется из таблицы;
3. Таблица обновляется.

---

**Ограничения:**

* кнопка `disabled`, если:

  * нет выбранной строки;
* после удаления интерфейс корректно обновляется.

## 🔍 Фильтрация заказов

Необходимо добавить блок фильтрации:

### `Filter by Item`

**Значения:**

* `All`
* `Notebook`
* `Pens`
* `Printer Paper`

---

### Логика

* при выборе значения отображаются только соответствующие записи
* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by Item`

---

### 📌 Назначение

Фильтр позволяет отображать заказы в зависимости от типа товара.

---

## 📊 Значения товаров и их смысл

| Товар | Значение | Когда используется |
| ----- | -------- | ------------------ |
| `Notebook` | Блокноты | заявки на канцелярию для записей |
| `Pens` | Ручки | заявки на пишущие принадлежности |
| `Printer Paper` | Бумага для принтера | заявки на расходные материалы |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Item
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `Notebook`

* отображаются только записи:

```json
{ "item": "Notebook" }
```

---

### `Pens`

* отображаются только:

```json
{ "item": "Pens" }
```

---

### `Printer Paper`

* отображаются только:

```json
{ "item": "Printer Paper" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No orders found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `New`, стала `Delivered`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий заказы в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление заказов

Для каждой записи:

* заполнить форму
* проверить `Create Order.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Одобрение заказов на бумагу для принтера

* найти заказы с `Item = Printer Paper`

* для каждой найденной записи:

  * выбрать строку
  * проверить доступность `Approve Order`
  * изменить статус на:

```json
{ "status": "Approved" }
```

---

### 4. Доставка одобренных заказов

* для каждой записи со статусом `Approved` и `Item = Printer Paper`:

  * выбрать строку
  * проверить доступность `Deliver Order`
  * изменить статус на:

```json
{ "status": "Delivered" }
```

---

### 5. Использование фильтра

* выбрать `Filter by Item = Printer Paper`
* убедиться, что отображаются только нужные записи

---

### 6. Поиск

* выполнить поиск по имени сотрудника (например: `"Ivan"`)
* проверить корректность результата

---

### 7. Проверка итогового состояния

* проверить:

  * все заказы добавлены (10 записей)
  * заказы на `Printer Paper` имеют статус `"Delivered"`
