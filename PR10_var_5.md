## 📖 Ситуация

После запуска базового портала регистрации заявок компания **SecureAccess Pro** столкнулась с необходимостью управления жизненным циклом запросов на доступ:

* необходимо подтверждать заявки после проверки полномочий;
* отклонять необоснованные или ошибочные запросы;
* отслеживать статус выполнения заявок.

В связи с этим было принято решение:

* расширить функциональность веб-приложения;
* реализовать более сложный сценарий автоматизации с использованием **Puzzle RPA**.

## 🔘 Новые кнопки управления

---

### `Approve Access`

**Назначение:**
Подтверждение заявки на доступ

**Логика:**

```json
{ "status": "Approved" }
```

**Требования:**

* доступна только если `Status ≠ "Approved"`
* при нажатии изменяет статус выбранной записи на `"Approved"`

---

### `Reject Access`

**Назначение:**
Отклонение заявки на доступ

**Логика:**

```json
{ "status": "Rejected" }
```

**Требования:**

* доступна только если `Status ≠ "Rejected"`
* при нажатии изменяет статус выбранной записи на `"Rejected"`

---

## ⚙️ Управление состоянием элементов (`disabled`)

---

### Кнопки `Approve Access`, `Reject Access`

Должны быть:

* `disabled = true`, если:

  * ни одна строка не выбрана

* `disabled = false`, если:

  * выбрана строка таблицы

---

### Дополнительные условия

* `Approve Access` должна быть неактивна, если:

```json
{ "status": "Approved" }
```

* `Reject Access` должна быть неактивна, если:

```json
{ "status": "Rejected" }
```

## 🔘 Детализация кнопок управления

---

### `Approve Access`

**Назначение:**
Подтверждение заявки после проверки полномочий сотрудника.

**Бизнес-смысл:**
Заявка проверена, доступ предоставлен сотруднику.

---

**Когда используется:**

* администратор проверил обоснование доступа;
* сотрудник имеет соответствующие полномочия.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус заявки:

   * если `Status = Approved` → операция запрещена;
   * если `Status ≠ Approved` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "Approved" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно подтвердить уже обработанную заявку;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Approved` или `Rejected`.

---

### `Reject Access`

**Назначение:**
Отклонение заявки на доступ.

**Бизнес-смысл:**
Заявка не соответствует политике безопасности или не обоснована.

---

**Когда используется:**

* недостаточное обоснование доступа;
* сотрудник не имеет полномочий для запрашиваемого уровня.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. Проверяется статус:

   * допускается только `Pending`.
3. При нажатии:

   * статус изменяется на:

```json
{ "status": "Rejected" }
```

4. Таблица обновляется.

---

**Ограничения:**

* повторное отклонение невозможно;
* кнопка `disabled`, если:

  * нет выбранной строки;
  * статус уже обработан.

## 🔍 Фильтрация заявок

Необходимо добавить блок фильтрации:

### `Filter by Access Level`

**Значения:**

* `All`
* `User`
* `Manager`
* `Admin`

---

### Логика

* при выборе значения отображаются только соответствующие записи
* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by Access Level`

---

### 📌 Назначение

Фильтр позволяет отображать заявки в зависимости от запрашиваемого уровня доступа.

---

## 📊 Значения уровней доступа и их смысл

| Уровень | Значение | Когда назначается |
| ------- | -------- | ----------------- |
| `User` | Базовый доступ | для стандартных операций |
| `Manager` | Расширенный доступ | для руководителей подразделений |
| `Admin` | Полный доступ | для администраторов систем |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Access Level
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `User`

* отображаются только записи:

```json
{ "access_level": "User" }
```

---

### `Manager`

* отображаются только:

```json
{ "access_level": "Manager" }
```

---

### `Admin`

* отображаются только:

```json
{ "access_level": "Admin" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No requests found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Pending`, стала `Approved`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий заявки в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление заявок

Для каждой записи:

* заполнить форму
* проверить `Submit Request.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Обработка заявок с уровнем Admin

* найти заявки с `Access Level = Admin`

* для каждой найденной заявки:

  * выбрать строку
  * проверить доступность `Approve Access`
  * изменить статус на:

```json
{ "status": "Approved" }
```

---

### 4. Использование фильтра

* выбрать `Filter by Access Level = Admin`
* убедиться, что отображаются только нужные записи

---

### 5. Поиск

* выполнить поиск по имени сотрудника (например: `"Ivan"`)
* проверить корректность результата

---

### 6. Отклонение заявки

* выбрать любую запись со статусом `Pending`
* нажать `Reject Access`
* убедиться, что статус изменился на `"Rejected"`

---

### 7. Проверка итогового состояния

* проверить:

  * все заявки добавлены (10 записей)
  * заявки с `Access Level = Admin` имеют статус `"Approved"`
  * одна заявка отклонена
