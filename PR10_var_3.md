# **Задание 1. Расширение функциональности веб-приложения**

Необходимо доработать существующее веб-приложение, добавив новые элементы управления и логику обработки багов.

**Каждый элемент интерфейса должен иметь уникальный `id`**

---

## 🔘 Новые кнопки управления

---

### `Assign Developer`

**Назначение:**
Назначение разработчика на исправление бага

**Логика:**

* открывает диалог выбора или автоматически присваивает разработчика
* записывает имя разработчика в запись

---

### `Mark Fixed`

**Назначение:**
Отметка бага как исправленного

**Логика:**

```json
{ "status": "Fixed" }
```

**Требования:**

* доступна только если `Status ≠ "Fixed"`
* при нажатии изменяет статус выбранной записи на `"Fixed"`

---

## ⚙️ Управление состоянием элементов (`disabled`)

---

### Кнопки `Assign Developer`, `Mark Fixed`

Должны быть:

* `disabled = true`, если:

  * ни одна строка не выбрана

* `disabled = false`, если:

  * выбрана строка таблицы

---

### Дополнительные условия

* `Mark Fixed` должна быть неактивна, если:

```json
{ "status": "Fixed" }
```

---

## 🔘 Детализация кнопок управления

---

### `Assign Developer`

**Назначение:**
Назначение ответственного разработчика на исправление бага.

**Бизнес-смысл:**
Баг передан конкретному специалисту для решения.

---

**Когда используется:**

* баг прошёл первичную фильтрацию;
* требуется ручное назначение исполнителя.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. При нажатии:

   * открывается окно выбора или автоматически назначается разработчик;
   * в запись добавляется поле `Assigned To`.
3. Таблица обновляется.

---

**Ограничения:**

* кнопка `disabled`, если:

  * нет выбранной строки;
* после назначения интерфейс корректно обновляется.

---

### `Mark Fixed`

**Назначение:**
Отметка бага как исправленного.

**Бизнес-смысл:**
Проблема устранена, баг переводится в архив.

---

**Когда используется:**

* разработчик устранил ошибку;
* тестировщик подтвердил исправление.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус бага:

   * если `Status = Fixed` → операция запрещена;
   * если `Status ≠ Fixed` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "Fixed" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно закрыть уже исправленный баг;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Fixed`.

## 🔍 Фильтрация багов

Необходимо добавить блок фильтрации:

### `Filter by Severity`

**Значения:**

* `All`
* `Low`
* `Medium`
* `Critical`

---

### Логика

* при выборе значения отображаются только соответствующие записи
* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by Severity`

---

### 📌 Назначение

Фильтр позволяет отображать баги в зависимости от их уровня критичности.

---

## 📊 Значения критичности и их смысл

| Критичность | Значение | Когда назначается |
| ----------- | -------- | ----------------- |
| `Low` | Низкая критичность | при незначительных проблемах |
| `Medium` | Средняя критичность | при стандартных ошибках |
| `Critical` | Высокая критичность | при критических сбоях |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Severity
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `Low`

* отображаются только записи:

```json
{ "severity": "Low" }
```

---

### `Medium`

* отображаются только:

```json
{ "severity": "Medium" }
```

---

### `Critical`

* отображаются только:

```json
{ "severity": "Critical" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No bugs found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Open`, стала `Fixed`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

---

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий баги в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление багов

Для каждой записи:

* заполнить форму
* проверить `Submit Bug.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Назначение разработчиков

После добавления:

* для каждой записи в таблице:

  * выбрать строку
  * проверить доступность `Assign Developer`
  * назначить разработчика
  * убедиться, что поле `Assigned To` заполнено

---

### 4. Обработка багов с критическим приоритетом

* найти баги с `Severity = Critical`

* для каждой найденной записи:

  * выбрать строку
  * проверить доступность `Mark Fixed`
  * изменить статус на:

```json
{ "status": "Fixed" }
```

---

### 5. Использование фильтра

* выбрать `Filter by Severity = Critical`
* убедиться, что отображаются только нужные записи

---

### 6. Поиск

* выполнить поиск по заголовку (например: `"Login"`)
* проверить корректность результата

---

### 7. Проверка итогового состояния

* проверить:

  * все баги добавлены (10 записей)
  * у всех записей заполнено `Assigned To`
  * баги с `Severity = Critical` имеют статус `"Fixed"`
