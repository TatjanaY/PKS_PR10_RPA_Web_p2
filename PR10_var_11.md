# **Кейс: «Автоматизация расследования и закрытия инцидентов безопасности»**

---

## 📖 Ситуация

После запуска базового портала регистрации инцидентов компания **SecureIT Pro** столкнулась с необходимостью управления жизненным циклом расследований:

* необходимо начинать расследование после классификации инцидента;
* закрывать инциденты после устранения угрозы;
* отслеживать статус выполнения работ.

В связи с этим было принято решение:

* расширить функциональность веб-приложения;
* реализовать более сложный сценарий автоматизации с использованием **Puzzle RPA**.

# **Задание 1. Расширение функциональности веб-приложения**

Необходимо доработать существующее веб-приложение, добавив новые элементы управления и логику обработки инцидентов.

**Каждый элемент интерфейса должен иметь уникальный `id`**

---

## 🔘 Новые кнопки управления

---

### `Start Investigation`

**Назначение:**
Начало расследования инцидента

**Логика:**

```json
{ "status": "Investigating" }
```

**Требования:**

* доступна только если `Status = "Reported"`
* при нажатии изменяет статус выбранной записи на `"Investigating"`

---

### `Close Incident`

**Назначение:**
Закрытие инцидента после устранения угрозы

**Логика:**

```json
{ "status": "Resolved" }
```

**Требования:**

* доступна только если `Status ≠ "Resolved"`
* при нажатии изменяет статус выбранной записи на `"Resolved"`

---

## ⚙️ Управление состоянием элементов (`disabled`)

---

### Кнопки `Start Investigation`, `Close Incident`

Должны быть:

* `disabled = true`, если:

  * ни одна строка не выбрана

* `disabled = false`, если:

  * выбрана строка таблицы

---

### Дополнительные условия

* `Start Investigation` должна быть неактивна, если:

```json
{ "status": "Investigating" }
или
{ "status": "Resolved" }
```

* `Close Incident` должна быть неактивна, если:

```json
{ "status": "Resolved" }
```

## 🔘 Детализация кнопок управления

---

### `Start Investigation`

**Назначение:**
Начало расследования после классификации инцидента.

**Бизнес-смысл:**
Специалист по безопасности приступил к анализу инцидента.

---

**Когда используется:**

* инцидент прошёл первичную классификацию;
* определён уровень критичности.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус инцидента:

   * если `Status ≠ Reported` → операция запрещена;
   * если `Status = Reported` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "Investigating" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно начать уже расследуемый инцидент;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Investigating` или `Resolved`.

---

### `Close Incident`

**Назначение:**
Закрытие инцидента после устранения угрозы.

**Бизнес-смысл:**
Угроза нейтрализована, инцидент архивируется.

---

**Когда используется:**

* проведён полный анализ инцидента;
* приняты меры по устранению уязвимости.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. Проверяется статус:

   * допускается любой статус кроме `Resolved`.
3. При нажатии:

   * статус изменяется на:

```json
{ "status": "Resolved" }
```

4. Таблица обновляется.

---

**Ограничения:**

* нельзя закрыть уже закрытый инцидент;
* кнопка `disabled`, если:

  * нет выбранной строки;
  * статус уже `Resolved`.

## 🔍 Фильтрация инцидентов

Необходимо добавить блок фильтрации:

### `Filter by Severity`

**Значения:**

* `All`
* `Low`
* `Medium`
* `High`
* `Critical`

---

### Логика

* при выборе значения отображаются только соответствующие записи
* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by Severity`

---

### 📌 Назначение

Фильтр позволяет отображать инциденты в зависимости от уровня критичности.

---

## 📊 Значения критичности и их смысл

| Критичность | Значение | Когда назначается |
| ----------- | -------- | ----------------- |
| `Low` | Низкая критичность | при незначительных нарушениях |
| `Medium` | Средняя критичность | при стандартных инцидентах |
| `High` | Высокая критичность | при серьёзных угрозах |
| `Critical` | Критическая | при активных атаках |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Severity
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `Low`

* отображаются только записи:

```json
{ "severity": "Low" }
```

---

### `Medium`

* отображаются только:

```json
{ "severity": "Medium" }
```

---

### `High`

* отображаются только:

```json
{ "severity": "High" }
```

---

### `Critical`

* отображаются только:

```json
{ "severity": "Critical" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No incidents found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Reported`, стала `Resolved`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий инциденты в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление инцидентов

Для каждой записи:

* заполнить форму
* проверить `Report Incident.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Начало расследования всех инцидентов

После добавления:

* для каждой записи в таблице:

  * выбрать строку
  * проверить доступность `Start Investigation`
  * изменить статус на:

```json
{ "status": "Investigating" }
```

---

### 4. Закрытие инцидентов с критическим уровнем

* найти инциденты с `Severity = Critical`

* для каждой найденной записи:

  * выбрать строку
  * проверить доступность `Close Incident`
  * изменить статус на:

```json
{ "status": "Resolved" }
```

---

### 5. Использование фильтра

* выбрать `Filter by Severity = Critical`
* убедиться, что отображаются только нужные записи

---

### 6. Поиск

* выполнить поиск по заголовку (например: `"Unauthorized"`)
* проверить корректность результата

---

### 7. Проверка итогового состояния

* проверить:

  * все инциденты добавлены (10 записей)
  * все записи имеют статус `"Investigating"` или `"Resolved"`
  * инциденты с `Severity = Critical` имеют статус `"Resolved"`
