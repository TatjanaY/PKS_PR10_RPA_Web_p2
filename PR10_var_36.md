## 📖 Ситуация

После запуска базового портала регистрации компания **ConferenceHub Pro** столкнулась с необходимостью управления жизненным циклом заявок:

* необходимо согласовывать заявки спикеров после проверки квалификации;
* отклонять не соответствующие требованиям регистрации;
* удалять неактуальные или ошибочные записи.

В связи с этим было принято решение:

* расширить функциональность веб-приложения;
* реализовать более сложный сценарий автоматизации с использованием **Puzzle RPA**.

## 🔘 Новые кнопки управления

---

### `Approve Participation`

**Назначение:**
Согласование участия в конференции

**Логика:**

```json
{ "status": "Approved" }
```

**Требования:**

* доступна только если `Status = "Registered"`
* при нажатии изменяет статус выбранной записи на `"Approved"`

---

### `Reject Participation`

**Назначение:**
Отклонение участия в конференции

**Логика:**

```json
{ "status": "Rejected" }
```

**Требования:**

* доступна только если `Status = "Registered"`
* при нажатии изменяет статус выбранной записи на `"Rejected"`

---

### `Delete Registration`

**Назначение:**
Удаление регистрации из системы

**Логика:**

* удаляет выбранную запись из таблицы

**Требования:**

* доступна только если выбрана строка таблицы
* после удаления таблица обновляется

---

## 🔘 Детализация кнопок управления

---

### `Approve Participation`

**Назначение:**
Согласование участия после проверки квалификации.

**Бизнес-смысл:**
Заявка проверена, участник допущен к конференции.

---

**Когда используется:**

* организатор проверил соответствие требованиям;
* для спикеров — подтверждена квалификация.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус регистрации:

   * если `Status ≠ Registered` → операция запрещена;
   * если `Status = Registered` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "Approved" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно согласовать уже обработанную заявку;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Approved` или `Rejected`.

---

### `Reject Participation`

**Назначение:**
Отклонение участия в конференции.

**Бизнес-смысл:**
Заявка не соответствует требованиям или квотам.

---

**Когда используется:**

* недостаточная квалификация спикера;
* превышено количество участников.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. Проверяется статус:

   * допускается только `Registered`.
3. При нажатии:

   * статус изменяется на:

```json
{ "status": "Rejected" }
```

4. Таблица обновляется.

---

**Ограничения:**

* повторное отклонение невозможно;
* кнопка `disabled`, если:

  * нет выбранной строки;
  * статус уже обработан.

---

### `Delete Registration`

**Назначение:**
Удаление регистрации из системы.

**Бизнес-смысл:**
Регистрация более не актуальна, запись удаляется.

---

**Когда используется:**

* участник отменил заявку;
* запись была создана ошибочно.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. При нажатии:

   * запись удаляется из таблицы;
3. Таблица обновляется.

---

**Ограничения:**

* кнопка `disabled`, если:

  * нет выбранной строки;
* после удаления интерфейс корректно обновляется.

---

## 🔍 Детализация фильтрации `Filter by Type`

---

### 📌 Назначение

Фильтр позволяет отображать регистрации в зависимости от типа участия.

---

## 📊 Значения типов участия и их смысл

| Тип участия | Значение | Когда используется |
| ----------- | -------- | ------------------ |
| `Speaker` | Спикер | заявки на выступление с докладом |
| `Listener` | Слушатель | заявки на участие в качестве слушателя |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Type
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `Speaker`

* отображаются только записи:

```json
{ "type": "Speaker" }
```

---

### `Listener`

* отображаются только:

```json
{ "type": "Listener" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No registrations found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Registered`, стала `Approved`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий регистрации в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление регистраций

Для каждой записи:

* заполнить форму
* проверить `Register Participant.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Одобрение участников типа Speaker

* найти регистрации с `Participation Type = Speaker`

* для каждой найденной записи:

  * выбрать строку
  * проверить доступность `Approve Participation`
  * изменить статус на:

```json
{ "status": "Approved" }
```

---

### 4. Отклонение двух заявок

* выбрать две записи со статусом `Registered`
* нажать `Reject Participation` для каждой
* убедиться, что статус изменился на `"Rejected"`

---

### 5. Удаление записи

* выбрать любую запись
* нажать `Delete Registration`
* убедиться, что запись исчезла

---

### 6. Использование фильтра

* выбрать `Filter by Type = Speaker`
* убедиться, что отображаются только нужные записи

---

### 7. Проверка итогового состояния

* проверить:

  * все регистрации добавлены (10 записей)
  * регистрации типа `Speaker` имеют статус `"Approved"`
  * две заявки отклонены
  * одна запись удалена
