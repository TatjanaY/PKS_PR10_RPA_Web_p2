## 📖 Ситуация

После запуска базового портала регистрации студентов компания **EduPlatform Pro** столкнулась с необходимостью управления жизненным циклом заявок:

* отслеживать статус выполнения заявок.

В связи с этим было принято решение:

* расширить функциональность веб-приложения;
* реализовать более сложный сценарий автоматизации с использованием **Puzzle RPA**.

## 🔘 Новые кнопки управления

---

### `Approve Student`

**Назначение:**
Подтверждение заявки студента на курс

**Логика:**

```json
{ "status": "Approved" }
```

**Требования:**

* доступна только если `Status = "Registered"`
* при нажатии изменяет статус выбранной записи на `"Approved"`

---

### `Reject Student`

**Назначение:**
Отклонение заявки студента

**Логика:**

```json
{ "status": "Rejected" }
```

**Требования:**

* доступна только если `Status = "Registered"`
* при нажатии изменяет статус выбранной записи на `"Rejected"`

---

## ⚙️ Управление состоянием элементов (`disabled`)

---

### Кнопки `Approve Student`, `Reject Student`

Должны быть:

* `disabled = true`, если:

  * ни одна строка не выбрана

* `disabled = false`, если:

  * выбрана строка таблицы

### Дополнительные условия

```json

```

* `Reject Student` должна быть неактивна, если:

```json
{ "status": "Approved" }

```

## 🔘 Детализация кнопок управления

---

### `Approve Student`

**Назначение:**
Подтверждение заявки после проверки данных студента.

**Бизнес-смысл:**
Заявка проверена, студент допущен к обучению.

---

**Когда используется:**

* администратор проверил корректность данных;
* студент соответствует требованиям курса.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус заявки:

   * если `Status ≠ Registered` → операция запрещена;
   * если `Status = Registered` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "Approved" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно подтвердить уже обработанную заявку;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Approved` или `Rejected`.

---

### `Reject Student`

**Назначение:**
Отклонение заявки студента.

**Бизнес-смысл:**
Заявка не соответствует требованиям или содержит ошибки.

---

**Когда используется:**

* некорректные данные студента;
* заявка не соответствует требованиям курса.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. Проверяется статус:

   * допускается только `Registered`.
3. При нажатии:

   * статус изменяется на:

```json
{ "status": "Rejected" }
```

4. Таблица обновляется.

---

**Ограничения:**

* повторное отклонение невозможно;
* кнопка `disabled`, если:

  * нет выбранной строки;
  * статус уже обработан.

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Course
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `Python`

* отображаются только записи:

```json
{ "course": "Python" }
```

---

### `Data Science`

* отображаются только:

```json
{ "course": "Data Science" }
```

---

### `Web Development`

* отображаются только:

```json
{ "course": "Web Development" }
```

---

### `DevOps`

* отображаются только:

```json
{ "course": "DevOps" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No registrations found
```

---

### 5. Взаимодействие с другими элементами

---

* запись может "исчезнуть" из текущего фильтра
  (например, была `Registered`, стала `Approved`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий заявки в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление заявок

Для каждой записи:

* заполнить форму
* проверить `Register Student.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Обработка заявок курса Python

* найти заявки с `Course = Python`

* для каждой найденной заявки:

  * выбрать строку
  * проверить доступность `Approve Student`
  * изменить статус на:

```json
{ "status": "Approved" }
```

---

### 4. Использование фильтра

* выбрать `Filter by Course = Python`
* убедиться, что отображаются только нужные записи

---

### 5. Поиск

* выполнить поиск по имени студента (например: `"Ivan"`)
* проверить корректность результата

---

### 6. Отклонение заявки

* выбрать любую запись со статусом `Registered`
* нажать `Reject Student`
* убедиться, что статус изменился на `"Rejected"`

---

### 7. Проверка итогового состояния

* проверить:

  * все заявки добавлены (10 записей)
  * заявки курса `Python` имеют статус `"Approved"`
  * одна заявка отклонена
