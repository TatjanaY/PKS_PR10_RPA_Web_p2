## 🔘 Новые кнопки управления

---

### `Finish Meeting`

**Назначение:**
Отметка о завершении встречи

**Логика:**

```json
{ "status": "Finished" }
```

**Требования:**

* доступна только если `Status ≠ "Finished"`
* при нажатии изменяет статус выбранной записи на `"Finished"`

---

### `Cancel Booking`

**Назначение:**
Отмена бронирования комнаты

**Логика:**

* удаляет выбранную запись из таблицы после подтверждения

**Требования:**

* доступна только если выбрана строка таблицы
* после удаления таблица обновляется

---

## ⚙️ Управление состоянием элементов (`disabled`)

---

### Кнопки `Finish Meeting`, `Cancel Booking`

Должны быть:

* `disabled = true`, если:

  * ни одна строка не выбрана

* `disabled = false`, если:

  * выбрана строка таблицы

---

### Дополнительные условия

* `Finish Meeting` должна быть неактивна, если:

```json
{ "status": "Finished" }
```

## 🔘 Детализация кнопок управления

---

### `Finish Meeting`

**Назначение:**
Отметка о завершении встречи после её окончания.

**Бизнес-смысл:**
Встреча завершена, комната освобождена для следующего бронирования.

---

**Когда используется:**

* встреча фактически завершилась;
* участники покинули помещение.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус бронирования:

   * если `Status = Finished` → операция запрещена;
   * если `Status ≠ Finished` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "Finished" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно завершить уже обработанную встречу;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Finished`.

---

### `Cancel Booking`

**Назначение:**
Отмена бронирования комнаты.

**Бизнес-смысл:**
Бронирование более не актуально, комната освобождается.

---

**Когда используется:**

* организатор отменил встречу;
* бронирование было создано ошибочно.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. При нажатии:

   * запись удаляется из таблицы;
3. Таблица обновляется.

---

**Ограничения:**

* кнопка `disabled`, если:

  * нет выбранной строки;
* после удаления интерфейс корректно обновляется.

---

### Логика

* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by Room`

---

### 📌 Назначение

Фильтр позволяет отображать бронирования в зависимости от выбранной комнаты.

---

## 📊 Значения комнат и их смысл

| Комната | Значение | Когда используется |
| ------- | -------- | ------------------ |
| `Alpha` | Основная переговорная | для крупных встреч |
| `Beta` | Средняя переговорная | для командных совещаний |
| `Gamma` | Малая переговорная | для индивидуальных встреч |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Room
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `Alpha`

* отображаются только записи:

```json
{ "room": "Alpha" }
```

---

### `Beta`

* отображаются только:

```json
{ "room": "Beta" }
```

---

### `Gamma`

* отображаются только:

```json
{ "room": "Gamma" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No bookings found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Booked`, стала `Finished`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий бронирования в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление бронирований

Для каждой записи:

* заполнить форму
* проверить `Add Booking.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Завершение встреч в комнате Alpha

* найти бронирования с `Room = Alpha`

* для каждой найденной записи:

  * выбрать строку
  * проверить доступность `Finish Meeting`
  * изменить статус на:

```json
{ "status": "Finished" }
```

---

### 4. Использование фильтра

* выбрать `Filter by Room = Alpha`
* убедиться, что отображаются только нужные записи

---

### 5. Поиск

* выполнить поиск по теме встречи (например: `"Project"`)
* проверить корректность результата

---

### 6. Удаление записи

* выбрать любую запись
* нажать `Cancel Booking`
* убедиться, что запись исчезла

---

### 7. Проверка итогового состояния

* проверить:

  * все бронирования добавлены (10 записей)
  * бронирования комнаты `Alpha` имеют статус `"Finished"`
  * одна запись удалена
