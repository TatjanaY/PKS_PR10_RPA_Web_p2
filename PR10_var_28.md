## 📖 Ситуация

После запуска базового портала регистрации заявок компания **MaintenancePro** столкнулась с необходимостью управления жизненным циклом сервисных работ:

* необходимо начинать обслуживание после классификации заявки;
* завершать работы после устранения неисправности;
* отслеживать статус выполнения заявок.

В связи с этим было принято решение:

* расширить функциональность веб-приложения;
* реализовать более сложный сценарий автоматизации с использованием **Puzzle RPA**.

## 🔘 Новые кнопки управления

---

### `Start Service`

**Назначение:**
Начало сервисного обслуживания

**Логика:**

```json
{ "status": "In Service" }
```

**Требования:**

* доступна только если `Status ≠ "Completed"`
* при нажатии изменяет статус выбранной записи на `"In Service"`

---

### `Complete Service`

**Назначение:**
Завершение сервисного обслуживания

**Логика:**

```json
{ "status": "Completed" }
```

**Требования:**

* доступна только если `Status = "In Service"`
* при нажатии изменяет статус выбранной записи на `"Completed"`

---

### `Delete Request`

**Назначение:**
Удаление заявки из системы

**Логика:**

* удаляет выбранную запись из таблицы

**Требования:**

* доступна только если выбрана строка таблицы
* после удаления таблица обновляется

---

## ⚙️ Управление состоянием элементов (`disabled`)

---

### Кнопки `Start Service`, `Complete Service`, `Delete Request`

Должны быть:

* `disabled = true`, если:

  * ни одна строка не выбрана

* `disabled = false`, если:

  * выбрана строка таблицы

---

### Дополнительные условия

* `Complete Service` должна быть неактивна, если:

```json
{ "status": "Completed" }
```

## 🔘 Детализация кнопок управления

---

### `Start Service`

**Назначение:**
Начало обслуживания после классификации заявки.

**Бизнес-смысл:**
Техник приступил к выполнению сервисных работ.

---

**Когда используется:**

* заявка прошла первичную диагностику;
* определён тип работ и приоритет.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус заявки:

   * если `Status = Completed` → операция запрещена;
   * если `Status ≠ Completed` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "In Service" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно начать уже выполняемую заявку;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Completed`.

---

### `Complete Service`

**Назначение:**
Завершение сервисного обслуживания.

**Бизнес-смысл:**
Работы выполнены, оборудование готово к эксплуатации.

---

**Когда используется:**

* техник завершил обслуживание;
* проведено тестирование оборудования.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. Проверяется статус:

   * допускается только `In Service`.
3. При нажатии:

   * статус изменяется на:

```json
{ "status": "Completed" }
```

4. Таблица обновляется.

---

**Ограничения:**

* нельзя завершить обслуживание, которое не было начато;
* кнопка `disabled`, если:

  * нет выбранной строки;
  * статус уже `Completed`.

---

### `Delete Request`

**Назначение:**
Удаление заявки из системы.

**Бизнес-смысл:**
Заявка более не актуальна, запись удаляется.

---

**Когда используется:**

* заявка отменена;
* запись была создана ошибочно.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. При нажатии:

   * запись удаляется из таблицы;
3. Таблица обновляется.

---

**Ограничения:**

* кнопка `disabled`, если:

  * нет выбранной строки;
* после удаления интерфейс корректно обновляется.

## 🔍 Фильтрация заявок

Необходимо добавить блок фильтрации:

### `Filter by Service Type`

**Значения:**

* `All`
* `Inspection`
* `Repair`
* `Replacement`

---

### Логика

* при выборе значения отображаются только соответствующие записи
* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by Service Type`

---

### 📌 Назначение

Фильтр позволяет отображать заявки в зависимости от типа сервисных работ.

---

## 📊 Значения типов работ и их смысл

| Тип работ | Значение | Когда используется |
| --------- | -------- | ------------------ |
| `Inspection` | Инспекция оборудования | плановые проверки |
| `Repair` | Ремонт | устранение неисправностей |
| `Replacement` | Замена компонентов | установка новых деталей |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Service Type
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `Inspection`

* отображаются только записи:

```json
{ "service_type": "Inspection" }
```

---

### `Repair`

* отображаются только:

```json
{ "service_type": "Repair" }
```

---

### `Replacement`

* отображаются только:

```json
{ "service_type": "Replacement" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No service requests found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Registered`, стала `Completed`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

---

Регистрация заявки → Registered
                ↓
           In Service

             Completed

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий заявки в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление заявок

Для каждой записи:

* заполнить форму
* проверить `Create Service Request.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Начало обслуживания заявок с высоким приоритетом

* найти заявки с `Priority = High`

* для каждой найденной записи:

  * выбрать строку
  * проверить доступность `Start Service`
  * изменить статус на:

```json
{ "status": "In Service" }
```

---

### 4. Завершение обслуживания заявок с высоким приоритетом

* для каждой записи со статусом `In Service` и `Priority = High`:

  * выбрать строку
  * проверить доступность `Complete Service`
  * изменить статус на:

```json
{ "status": "Completed" }
```

---

### 5. Использование фильтра

* выбрать `Filter by Service Type = Inspection`
* убедиться, что отображаются только нужные записи

---

### 6. Поиск

* выполнить поиск по идентификатору оборудования (например: `"EQP-001"`)
* проверить корректность результата

---

### 7. Проверка итогового состояния

* проверить:

  * все заявки добавлены (10 записей)
  * заявки с `Priority = High` имеют статус `"Completed"`
