## 📖 Ситуация

После запуска базового портала регистрации ошибок компания **BugTracker Pro** столкнулась с необходимостью управления жизненным циклом багов:

* необходимо начинать исправление ошибок после их классификации;
* закрывать баги после устранения проблемы;
* отслеживать статус выполнения работ.

В связи с этим было принято решение:

* расширить функциональность веб-приложения;
* реализовать более сложный сценарий автоматизации с использованием **Puzzle RPA**.

# **Задание 1. Расширение функциональности веб-приложения**

Необходимо доработать существующее веб-приложение, добавив новые элементы управления и логику обработки ошибок.

**Каждый элемент интерфейса должен иметь уникальный `id`**

---

## 🔘 Новые кнопки управления

---

### `Start Fix`

**Назначение:**
Начало исправления ошибки

**Логика:**

```json
{ "status": "In Progress" }
```

**Требования:**

* доступна только если `Status ≠ "Closed"`
* при нажатии изменяет статус выбранной записи на `"In Progress"`

---

### `Close Bug`

**Назначение:**
Закрытие ошибки после исправления

**Логика:**

```json
{ "status": "Closed" }
```

**Требования:**

* доступна только если `Status ≠ "Closed"`
* при нажатии изменяет статус выбранной записи на `"Closed"`

---

## ⚙️ Управление состоянием элементов (`disabled`)

---

### Кнопки `Start Fix`, `Close Bug`

Должны быть:

* `disabled = true`, если:

  * ни одна строка не выбрана

* `disabled = false`, если:

  * выбрана строка таблицы

---

### Дополнительные условия

* `Close Bug` должна быть неактивна, если:

```json
{ "status": "Closed" }
```

## 🔘 Детализация кнопок управления

---

### `Start Fix`

**Назначение:**
Начало исправления ошибки после классификации.

**Бизнес-смысл:**
Разработчик приступил к устранению бага.

---

**Когда используется:**

* ошибка прошла первичную классификацию;
* определён уровень критичности.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус ошибки:

   * если `Status = Closed` → операция запрещена;
   * если `Status ≠ Closed` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "In Progress" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно начать уже исправляемую ошибку;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Closed`.

---

### `Close Bug`

**Назначение:**
Закрытие ошибки после исправления.

**Бизнес-смысл:**
Проблема устранена, баг архивируется.

---

**Когда используется:**

* разработчик устранил ошибку;
* проведено тестирование исправления.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. Проверяется статус:

   * допускается любой статус кроме `Closed`.
3. При нажатии:

   * статус изменяется на:

```json
{ "status": "Closed" }
```

4. Таблица обновляется.

---

**Ограничения:**

* нельзя закрыть уже закрытый баг;
* кнопка `disabled`, если:

  * нет выбранной строки;
  * статус уже `Closed`.

## 🔍 Фильтрация ошибок

Необходимо добавить блок фильтрации:

### `Filter by Severity`

**Значения:**

* `All`
* `Minor`
* `Major`
* `Critical`

---

### Логика

* при выборе значения отображаются только соответствующие записи
* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by Severity`

---

### 📌 Назначение

Фильтр позволяет отображать ошибки в зависимости от уровня критичности.

---

## 📊 Значения критичности и их смысл

| Критичность | Значение | Когда назначается |
| ----------- | -------- | ----------------- |
| `Minor` | Низкая критичность | при незначительных проблемах |
| `Major` | Средняя критичность | при серьёзных ошибках |
| `Critical` | Высокая критичность | при критических сбоях |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Severity
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `Minor`

* отображаются только записи:

```json
{ "severity": "Minor" }
```

---

### `Major`

* отображаются только:

```json
{ "severity": "Major" }
```

---

### `Critical`

* отображаются только:

```json
{ "severity": "Critical" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No bugs found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Reported`, стала `Closed`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

---

Регистрация ошибки → Reported
                ↓
           In Progress
                ↓
              Closed

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий ошибки в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление ошибок

Для каждой записи:

* заполнить форму
* проверить `Add Bug.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Начало исправления ошибок с критическим уровнем

* найти ошибки с `Severity = Critical`

* для каждой найденной записи:

  * выбрать строку
  * проверить доступность `Start Fix`
  * изменить статус на:

```json
{ "status": "In Progress" }
```

---

### 4. Закрытие ошибок с критическим уровнем

* для каждой ошибки со статусом `In Progress` и `Severity = Critical`:

  * выбрать строку
  * проверить доступность `Close Bug`
  * изменить статус на:

```json
{ "status": "Closed" }
```

---

### 5. Использование фильтра

* выбрать `Filter by Severity = Critical`
* убедиться, что отображаются только нужные записи

---

### 6. Поиск

* выполнить поиск по заголовку (например: `"Login"`)
* проверить корректность результата

---

### 7. Проверка итогового состояния

* проверить:

  * все ошибки добавлены (10 записей)
  * ошибки с `Severity = Critical` имеют статус `"Closed"`
