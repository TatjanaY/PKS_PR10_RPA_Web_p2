## 📖 Ситуация

После запуска базового портала регистрации клиентов компания **CRMPro Solutions** столкнулась с необходимостью управления жизненным циклом клиентов:

* необходимо согласовывать клиентов после проверки данных;
* удалять неактуальные или ошибочные записи;
* отслеживать статус выполнения заявок.

В связи с этим было принято решение:

* расширить функциональность веб-приложения;
* реализовать более сложный сценарий автоматизации с использованием **Puzzle RPA**.

# **Задание 1. Расширение функциональности веб-приложения**

Необходимо доработать существующее веб-приложение, добавив новые элементы управления и логику обработки клиентов.

**Каждый элемент интерфейса должен иметь уникальный `id`**

---

## 🔘 Новые кнопки управления

---

### `Approve Client`

**Назначение:**
Согласование клиента в системе

**Логика:**

```json
{ "status": "Approved" }
```

**Требования:**

* доступна только если `Status = "New"`
* при нажатии изменяет статус выбранной записи на `"Approved"`

---

### `Delete Client`

**Назначение:**
Удаление клиента из системы

**Логика:**

* удаляет выбранную запись из таблицы

**Требования:**

* доступна только если выбрана строка таблицы
* после удаления таблица обновляется

---

## ⚙️ Управление состоянием элементов (`disabled`)

---

### Кнопки `Approve Client`, `Delete Client`

Должны быть:

* `disabled = true`, если:

  * ни одна строка не выбрана

* `disabled = false`, если:

  * выбрана строка таблицы

---

### Дополнительные условия

* `Approve Client` должна быть неактивна, если:

```json
{ "status": "Approved" }
```

---

## 🔘 Детализация кнопок управления

---

### `Approve Client`

**Назначение:**
Согласование клиента после проверки данных.

**Бизнес-смысл:**
Клиент проверен, доступ к системе предоставлен.

---

**Когда используется:**

* менеджер проверил корректность данных;
* клиент соответствует требованиям компании.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус клиента:

   * если `Status ≠ New` → операция запрещена;
   * если `Status = New` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "Approved" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно согласовать уже обработанного клиента;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Approved`.

---

### `Delete Client`

**Назначение:**
Удаление клиента из системы.

**Бизнес-смысл:**
Запись более не актуальна, клиент удаляется.

---

**Когда используется:**

* клиент отменил регистрацию;
* запись была создана ошибочно.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. При нажатии:

   * запись удаляется из таблицы;
3. Таблица обновляется.

---

**Ограничения:**

* кнопка `disabled`, если:

  * нет выбранной строки;
* после удаления интерфейс корректно обновляется.

## 🔍 Фильтрация клиентов

Необходимо добавить блок фильтрации:

### `Filter by Type`

**Значения:**

* `All`
* `Individual`
* `Business`

---

### Логика

* при выборе значения отображаются только соответствующие записи
* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by Type`

---

### 📌 Назначение

Фильтр позволяет отображать клиентов в зависимости от типа.

---

## 📊 Значения типов и их смысл

| Тип | Значение | Когда используется |
| --- | -------- | ------------------ |
| `Individual` | Индивидуальный клиент | заявки от частных лиц |
| `Business` | Бизнес-клиент | заявки от компаний |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Type
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `Individual`

* отображаются только записи:

```json
{ "type": "Individual" }
```

---

### `Business`

* отображаются только:

```json
{ "type": "Business" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No clients found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `New`, стала `Approved`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

---

Регистрация клиента → New

              Approved

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий клиентов в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление клиентов

Для каждой записи:

* заполнить форму
* проверить `Add Client.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Согласование клиентов типа Business

* найти клиентов с `Type = Business`

* для каждой найденной записи:

  * выбрать строку
  * проверить доступность `Approve Client`
  * изменить статус на:

```json
{ "status": "Approved" }
```

---

### 4. Использование фильтра

* выбрать `Filter by Type = Business`
* убедиться, что отображаются только нужные записи

---

### 5. Поиск

* выполнить поиск по имени клиента (например: `"Ivan"`)
* проверить корректность результата

---

### 6. Удаление записи

* выбрать любую запись
* нажать `Delete Client`
* убедиться, что запись исчезла

---

### 7. Проверка итогового состояния

* проверить:

  * все клиенты добавлены (10 записей)
  * клиенты типа `Business` имеют статус `"Approved"`
  * одна запись удалена
