# **Кейс: «Автоматизация согласования и отклонения заявок на оборудование»**

---

## 📖 Ситуация

После запуска базового портала регистрации заявок компания **OfficeEquip Pro** столкнулась с необходимостью управления жизненным циклом заявок:

* необходимо согласовывать заявки после проверки наличия оборудования;
* отклонять необоснованные или не соответствующие политике заявки;
* удалять неактуальные или ошибочные записи.

В связи с этим было принято решение:

* расширить функциональность веб-приложения;
* реализовать более сложный сценарий автоматизации с использованием **Puzzle RPA**.

## 🔘 Новые кнопки управления

---

### `Approve Request`

**Назначение:**
Согласование заявки на оборудование

**Логика:**

```json
{ "status": "Approved" }
```

**Требования:**

* доступна только если `Status = "Requested"`
* при нажатии изменяет статус выбранной записи на `"Approved"`

---

### `Reject Request`

**Назначение:**
Отклонение заявки на оборудование

**Логика:**

```json
{ "status": "Rejected" }
```

**Требования:**

* доступна только если `Status = "Requested"`
* при нажатии изменяет статус выбранной записи на `"Rejected"`

---

### `Delete Request`

**Назначение:**
Удаление заявки из системы

**Логика:**

* удаляет выбранную запись из таблицы

**Требования:**

* доступна только если выбрана строка таблицы
* после удаления таблица обновляется

---

## ⚙️ Управление состоянием элементов (`disabled`)

---

Должны быть:

* `disabled = true`, если:

  * ни одна строка не выбрана

* `disabled = false`, если:

  * выбрана строка таблицы

```json

```

* `Reject Request` должна быть неактивна, если:

```json
{ "status": "Approved" }
```

## 🔘 Детализация кнопок управления

---

### `Approve Request`

**Назначение:**
Согласование заявки после проверки наличия оборудования.

**Бизнес-смысл:**
Заявка проверена, оборудование зарезервировано для сотрудника.

---

**Когда используется:**

* администратор проверил наличие на складе;
* заявка соответствует политике компании.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус заявки:

   * если `Status ≠ Requested` → операция запрещена;
   * если `Status = Requested` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "Approved" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно согласовать уже обработанную заявку;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Approved` или `Rejected`.

---

### `Reject Request`

**Назначение:**
Отклонение заявки на оборудование.

**Бизнес-смысл:**
Заявка не соответствует политике или оборудованию нет в наличии.

---

**Когда используется:**

* недостаточное обоснование запроса;
* оборудование отсутствует на складе.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. Проверяется статус:

   * допускается только `Requested`.
3. При нажатии:

   * статус изменяется на:

```json
{ "status": "Rejected" }
```

4. Таблица обновляется.

---

**Ограничения:**

* повторное отклонение невозможно;
* кнопка `disabled`, если:

  * нет выбранной строки;
  * статус уже обработан.

---

### `Delete Request`

**Назначение:**
Удаление заявки из системы.

**Бизнес-смысл:**
Заявка более не актуальна, запись удаляется.

---

**Когда используется:**

* сотрудник отменил запрос;
* запись была создана ошибочно.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. При нажатии:

   * запись удаляется из таблицы;
3. Таблица обновляется.

---

**Ограничения:**

* кнопка `disabled`, если:

  * нет выбранной строки;
* после удаления интерфейс корректно обновляется.

## 🔍 Фильтрация заявок

Необходимо добавить блок фильтрации:

### `Filter by Priority`

**Значения:**

* `All`
* `Low`
* `Medium`
* `High`

---

### Логика

* при выборе значения отображаются только соответствующие записи
* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by Priority`

---

### 📌 Назначение

Фильтр позволяет отображать заявки в зависимости от приоритета.

---

## 📊 Значения приоритетов и их смысл

| Приоритет | Значение | Когда назначается |
| --------- | -------- | ----------------- |
| `Low` | Низкий приоритет | при плановых запросах |
| `Medium` | Средний приоритет | при стандартных потребностях |
| `High` | Высокий приоритет | при критических потребностях |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Priority
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `Low`

* отображаются только записи:

```json
{ "priority": "Low" }
```

---

### `Medium`

* отображаются только:

```json
{ "priority": "Medium" }
```

---

### `High`

* отображаются только:

```json
{ "priority": "High" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No requests found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Requested`, стала `Approved`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий заявки в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление заявок

Для каждой записи:

* заполнить форму
* проверить `Submit Request.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Одобрение заявок с высоким приоритетом

* найти заявки с `Priority = High`

* для каждой найденной заявки:

  * выбрать строку
  * проверить доступность `Approve Request`
  * изменить статус на:

```json
{ "status": "Approved" }
```

---

### 4. Отклонение двух заявок

* выбрать две записи со статусом `Requested`
* нажать `Reject Request` для каждой
* убедиться, что статус изменился на `"Rejected"`

---

### 5. Удаление записи

* выбрать любую запись
* нажать `Delete Request`
* убедиться, что запись исчезла

---

### 6. Использование фильтра

* выбрать `Filter by Priority = High`
* убедиться, что отображаются только нужные записи

---

### 7. Проверка итогового состояния

* проверить:

  * все заявки добавлены (10 записей)
  * заявки с `Priority = High` имеют статус `"Approved"`
  * две заявки отклонены
  * одна запись удалена
