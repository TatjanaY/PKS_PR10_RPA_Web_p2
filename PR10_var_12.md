## 📖 Ситуация

После запуска базового портала регистрации заявок компания **EduCorp Pro** столкнулась с необходимостью управления жизненным циклом заявок на обучение:

* отклонять необоснованные или не соответствующие политике заявки;
* отслеживать статус выполнения заявок.

В связи с этим было принято решение:

* расширить функциональность веб-приложения;
* реализовать более сложный сценарий автоматизации с использованием **Puzzle RPA**.

## 🔘 Новые кнопки управления

---

### `Approve Training`

**Назначение:**
Согласование заявки на обучение

**Логика:**

```json
{ "status": "Approved" }
```

**Требования:**

* доступна только если `Status = "Pending Approval"`
* при нажатии изменяет статус выбранной записи на `"Approved"`

---

### `Reject Training`

**Назначение:**
Отклонение заявки на обучение

**Логика:**

```json
{ "status": "Rejected" }
```

**Требования:**

* доступна только если `Status = "Pending Approval"`
* при нажатии изменяет статус выбранной записи на `"Rejected"`

## 🔘 Детализация кнопок управления

---

### `Approve Training`

**Назначение:**
Согласование заявки после проверки бюджета и соответствия политике обучения.

**Бизнес-смысл:**
Заявка проверена, обучение согласовано для сотрудника.

---

**Когда используется:**

* руководитель проверил обоснование обучения;
* бюджет позволяет оплатить курс.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус заявки:

   * если `Status ≠ Pending Approval` → операция запрещена;
   * если `Status = Pending Approval` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "Approved" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно согласовать уже обработанную заявку;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Approved` или `Rejected`.

---

### `Reject Training`

**Назначение:**
Отклонение заявки на обучение.

**Бизнес-смысл:**
Заявка не соответствует политике обучения или бюджетным ограничениям.

---

**Когда используется:**

* недостаточное обоснование обучения;
* превышен бюджет подразделения.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. Проверяется статус:

   * допускается только `Pending Approval`.
3. При нажатии:

   * статус изменяется на:

```json
{ "status": "Rejected" }
```

4. Таблица обновляется.

---

**Ограничения:**

* повторное отклонение невозможно;
* кнопка `disabled`, если:

  * нет выбранной строки;
  * статус уже обработан.

## 🔍 Фильтрация заявок

Необходимо добавить блок фильтрации:

### `Filter by Format`

**Значения:**

* `All`
* `Online`
* `Offline`
* `Hybrid`

---

### Логика

* при выборе значения отображаются только соответствующие записи
* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by Format`

---

### 📌 Назначение

Фильтр позволяет отображать заявки в зависимости от формата обучения.

---

## 📊 Значения форматов и их смысл

| Формат | Значение | Когда используется |
| ------ | -------- | ------------------ |
| `Online` | Дистанционное обучение | заявки на онлайн-курсы |
| `Offline` | Очное обучение | заявки на очные тренинги |
| `Hybrid` | Смешанный формат | заявки на комбинированные программы |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Format
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `Online`

* отображаются только записи:

```json
{ "format": "Online" }
```

---

### `Offline`

* отображаются только:

```json
{ "format": "Offline" }
```

---

### `Hybrid`

* отображаются только:

```json
{ "format": "Hybrid" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No training requests found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Pending Approval`, стала `Approved`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий заявки в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление заявок

Для каждой записи:

* заполнить форму
* проверить `Submit Training Request.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Согласование заявок формата Online

* найти заявки с `Format = Online`

* для каждой найденной заявки:

  * выбрать строку
  * проверить доступность `Approve Training`
  * изменить статус на:

```json
{ "status": "Approved" }
```

---

### 4. Использование фильтра

* выбрать `Filter by Format = Online`
* убедиться, что отображаются только нужные записи

---

### 5. Поиск

* выполнить поиск по имени сотрудника (например: `"Ivan"`)
* проверить корректность результата

---

### 6. Отклонение заявки

* выбрать любую запись со статусом `Pending Approval`
* нажать `Reject Training`
* убедиться, что статус изменился на `"Rejected"`

---

### 7. Проверка итогового состояния

* проверить:

  * все заявки добавлены (10 записей)
  * заявки формата `Online` имеют статус `"Approved"`
  * одна заявка отклонена
