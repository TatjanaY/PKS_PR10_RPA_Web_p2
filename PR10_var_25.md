## 📖 Ситуация

После запуска базового портала регистрации компания **EventHR Pro** столкнулась с необходимостью управления жизненным циклом заявок:

* необходимо согласовывать заявки после проверки доступности мест;
* отклонять неактуальные или не соответствующие требованиям заявки;
* удалять неактуальные или ошибочные записи.

В связи с этим было принято решение:

* расширить функциональность веб-приложения;
* реализовать более сложный сценарий автоматизации с использованием **Puzzle RPA**.

## 🔘 Новые кнопки управления

---

### `Approve Registration`

**Назначение:**
Согласование регистрации на мероприятие

**Логика:**

```json
{ "status": "Approved" }
```

**Требования:**

* доступна только если `Status = "Registered"`
* при нажатии изменяет статус выбранной записи на `"Approved"`

---

### `Reject Registration`

**Назначение:**
Отклонение регистрации на мероприятие

**Логика:**

```json
{ "status": "Rejected" }
```

**Требования:**

* доступна только если `Status = "Registered"`
* при нажатии изменяет статус выбранной записи на `"Rejected"`

---

### `Delete Registration`

**Назначение:**
Удаление регистрации из системы

**Логика:**

* удаляет выбранную запись из таблицы

**Требования:**

* доступна только если выбрана строка таблицы
* после удаления таблица обновляется

---

  * выбрана строка таблицы

## 🔘 Детализация кнопок управления

---

### `Approve Registration`

**Назначение:**
Согласование регистрации после проверки доступности мест.

**Бизнес-смысл:**
Заявка проверена, сотрудник допущен к мероприятию.

---

**Когда используется:**

* администратор проверил корректность данных;
* есть свободные места на мероприятии.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус регистрации:

   * если `Status ≠ Registered` → операция запрещена;
   * если `Status = Registered` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "Approved" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно согласовать уже обработанную заявку;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Approved` или `Rejected`.

---

### `Reject Registration`

**Назначение:**
Отклонение регистрации на мероприятие.

**Бизнес-смысл:**
Заявка не соответствует требованиям или мест нет.

---

**Когда используется:**

* недостаточное обоснование участия;
* мероприятие переполнено.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. Проверяется статус:

   * допускается только `Registered`.
3. При нажатии:

   * статус изменяется на:

```json
{ "status": "Rejected" }
```

4. Таблица обновляется.

---

**Ограничения:**

* повторное отклонение невозможно;
* кнопка `disabled`, если:

  * нет выбранной строки;
  * статус уже обработан.

---

### `Delete Registration`

**Назначение:**
Удаление регистрации из системы.

**Бизнес-смысл:**
Регистрация более не актуальна, запись удаляется.

---

**Когда используется:**

* сотрудник отменил заявку;
* запись была создана ошибочно.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. При нажатии:

   * запись удаляется из таблицы;
3. Таблица обновляется.

---

**Ограничения:**

* кнопка `disabled`, если:

  * нет выбранной строки;
* после удаления интерфейс корректно обновляется.

## 🔍 Детализация фильтрации `Filter by Event Type`

---

### 📌 Назначение

Фильтр позволяет отображать регистрации в зависимости от типа мероприятия.

---

## 📊 Значения мероприятий и их смысл

| Мероприятие | Значение | Когда используется |
| ----------- | -------- | ------------------ |
| `Team Building` | Командообразование | заявки на тимбилдинги |
| `Workshop` | Практический воркшоп | заявки на обучение |
| `Conference` | Конференция | заявки на профессиональные события |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Event Type
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `Team Building`

* отображаются только записи:

```json
{ "event_type": "Team Building" }
```

---

### `Workshop`

* отображаются только:

```json
{ "event_type": "Workshop" }
```

---

### `Conference`

* отображаются только:

```json
{ "event_type": "Conference" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No registrations found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Registered`, стала `Approved`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий регистрации в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление регистраций

Для каждой записи:

* заполнить форму
* проверить `Submit Registration.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Одобрение регистраций на конференции

* найти регистрации с `Event Type = Conference`

* для каждой найденной записи:

  * выбрать строку
  * проверить доступность `Approve Registration`
  * изменить статус на:

```json
{ "status": "Approved" }
```

---

### 4. Использование фильтра

* выбрать `Filter by Event Type = Conference`
* убедиться, что отображаются только нужные записи

---

### 5. Поиск

* выполнить поиск по имени сотрудника (например: `"Ivan"`)
* проверить корректность результата

---

### 6. Отклонение регистрации

* выбрать любую запись со статусом `Registered`
* нажать `Reject Registration`
* убедиться, что статус изменился на `"Rejected"`

---

### 7. Удаление записи

* выбрать любую запись
* нажать `Delete Registration`
* убедиться, что запись исчезла

---

### 8. Проверка итогового состояния

* проверить:

  * все регистрации добавлены (10 записей)
  * регистрации на `Conference` имеют статус `"Approved"`
  * одна регистрация отклонена
  * одна запись удалена
