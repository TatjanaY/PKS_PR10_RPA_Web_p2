## 🔘 Новые кнопки управления

---

### `Confirm Booking`

**Назначение:**
Подтверждение бронирования

**Логика:**

```json
{ "status": "Confirmed" }
```

**Требования:**

* доступна только если `Status ≠ "Confirmed"`
* при нажатии изменяет статус выбранной записи на `"Confirmed"`

---

### `Cancel Booking`

**Назначение:**
Отмена бронирования

**Логика:**

* удаляет выбранную запись из таблицы

**Требования:**

* доступна только если выбрана строка таблицы
* после удаления таблица обновляется

---

## ⚙️ Управление состоянием элементов (`disabled`)

---

### Кнопки `Confirm Booking`, `Cancel Booking`

Должны быть:

* `disabled = true`, если:

  * ни одна строка не выбрана

* `disabled = false`, если:

  * выбрана строка таблицы

---

### Дополнительные условия

* `Confirm Booking` должна быть неактивна, если:

```json
{ "status": "Confirmed" }
```

## 🔘 Детализация кнопок управления

---

### `Confirm Booking`

**Назначение:**
Подтверждение бронирования после проверки расписания.

**Бизнес-смысл:**
Заявка проверена, комната зарезервирована за сотрудником.

---

**Когда используется:**

* администратор проверил отсутствие конфликтов;
* бронирование соответствует политике компании.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус заявки:

   * если `Status = Confirmed` → операция запрещена;
   * если `Status ≠ Confirmed` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "Confirmed" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно подтвердить уже обработанную заявку;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Confirmed`.

---

### `Cancel Booking`

**Назначение:**
Отмена бронирования.

**Бизнес-смысл:**
Заявка более не актуальна, комната освобождается.

---

**Когда используется:**

* сотрудник отменил встречу;
* бронирование было создано ошибочно.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. При нажатии:

   * запись удаляется из таблицы;
3. Таблица обновляется.

---

**Ограничения:**

* кнопка `disabled`, если:

  * нет выбранной строки;
* после удаления интерфейс корректно обновляется.

## 🔍 Фильтрация бронирований

Необходимо добавить блок фильтрации:

### `Filter by Status`

**Значения:**

* `All`
* `Pending`
* `Confirmed`

---

### Логика

* при выборе значения отображаются только соответствующие записи
* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by Status`

---

### 📌 Назначение

Фильтр позволяет отображать бронирования в зависимости от их текущего статуса.

---

## 📊 Значения статусов и их смысл

| Статус | Значение | Когда назначается |
| ------ | -------- | ----------------- |
| `Pending` | Заявка создана, но не подтверждена | при отправке формы |
| `Confirmed` | Бронирование подтверждено | при нажатии `Confirm Booking` |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Status
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `Pending`

* отображаются только записи:

```json
{ "status": "Pending" }
```

---

### `Confirmed`

* отображаются только:

```json
{ "status": "Confirmed" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No bookings found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Pending`, стала `Confirmed`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

---

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий бронирования в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление бронирований

Для каждой записи:

* заполнить форму
* проверить `Book Room.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Подтверждение бронирований

После добавления:

* для каждой записи в таблице:

  * выбрать строку
  * проверить доступность `Confirm Booking`
  * изменить статус на:

```json
{ "status": "Confirmed" }
```

---

### 4. Использование фильтра

* выбрать `Filter by Status = Confirmed`
* убедиться, что отображаются только нужные записи

---

### 5. Поиск

* выполнить поиск по имени сотрудника (например: `"Ivan"`)
* проверить корректность результата

---

### 6. Удаление записи

* выбрать любую запись
* нажать `Cancel Booking`
* убедиться, что запись исчезла

---

### 7. Проверка итогового состояния

* проверить:

  * все бронирования добавлены (10 записей)
  * все записи имеют статус `"Confirmed"`
  * одна запись удалена
