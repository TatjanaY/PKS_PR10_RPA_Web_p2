# **Кейс: «Автоматизация подтверждения и отмены бронирований парковочных мест»**

---

## 📖 Ситуация

После запуска базового портала регистрации бронирований компания **ParkingPro** столкнулась с необходимостью управления жизненным циклом заявок:

* необходимо подтверждать бронирования после проверки доступности мест;
* отменять неактуальные или ошибочные заявки;
* удалять неактуальные или ошибочные записи.

В связи с этим было принято решение:

* расширить функциональность веб-приложения;
* реализовать более сложный сценарий автоматизации с использованием **Puzzle RPA**.

Необходимо доработать существующее веб-приложение, добавив новые элементы управления и логику обработки бронирований.

## 🔘 Новые кнопки управления

---

### `Confirm Booking`

**Назначение:**
Подтверждение бронирования парковочного места

**Логика:**

```json
{ "status": "Confirmed" }
```

**Требования:**

* доступна только если `Status = "Booked"`
* при нажатии изменяет статус выбранной записи на `"Confirmed"`

---

### `Cancel Booking`

**Назначение:**
Отмена бронирования парковочного места

**Логика:**

```json
{ "status": "Cancelled" }
```

**Требования:**

* доступна только если `Status = "Booked"`
* при нажатии изменяет статус выбранной записи на `"Cancelled"`

---

### `Delete Booking`

**Назначение:**
Удаление бронирования из системы

**Логика:**

* удаляет выбранную запись из таблицы

**Требования:**

* доступна только если выбрана строка таблицы
* после удаления таблица обновляется

---

## ⚙️ Управление состоянием элементов (`disabled`)

---

### Кнопки `Confirm Booking`, `Cancel Booking`, `Delete Booking`

Должны быть:

* `disabled = true`, если:

  * ни одна строка не выбрана

* `disabled = false`, если:

  * выбрана строка таблицы

---

### Дополнительные условия

* `Confirm Booking` должна быть неактивна, если:

```json
{ "status": "Confirmed" }
или
{ "status": "Cancelled" }
```

* `Cancel Booking` должна быть неактивна, если:

```json
{ "status": "Confirmed" }
или
{ "status": "Cancelled" }
```

## 🔘 Детализация кнопок управления

---

### `Confirm Booking`

**Назначение:**
Подтверждение бронирования после проверки доступности мест.

**Бизнес-смысл:**
Заявка проверена, парковочное место зарезервировано за сотрудником.

---

**Когда используется:**

* администратор проверил корректность данных;
* есть свободные места в выбранной зоне.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус бронирования:

   * если `Status ≠ Booked` → операция запрещена;
   * если `Status = Booked` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "Confirmed" }
```

4. Таблица обновляется (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно подтвердить уже обработанную заявку;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Confirmed` или `Cancelled`.

---

### `Cancel Booking`

**Назначение:**
Отмена бронирования парковочного места.

**Бизнес-смысл:**
Заявка более не актуальна, место освобождается.

---

**Когда используется:**

* сотрудник отменил бронирование;
* запись была создана ошибочно.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. Проверяется статус:

   * допускается только `Booked`.
3. При нажатии:

   * статус изменяется на:

```json
{ "status": "Cancelled" }
```

4. Таблица обновляется.

---

**Ограничения:**

* повторная отмена невозможна;
* кнопка `disabled`, если:

  * нет выбранной строки;
  * статус уже обработан.

---

### `Delete Booking`

**Назначение:**
Удаление бронирования из системы.

**Бизнес-смысл:**
Бронирование более не актуально, запись удаляется.

---

**Когда используется:**

* сотрудник отменил заявку;
* запись была создана ошибочно.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. При нажатии:

   * запись удаляется из таблицы;
3. Таблица обновляется.

---

**Ограничения:**

* кнопка `disabled`, если:

  * нет выбранной строки;
* после удаления интерфейс корректно обновляется.

## 🔍 Фильтрация бронирований

**Значения:**

* `All`
* `A`
* `B`
* `C`

---

### Логика

* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by Zone`

---

### 📌 Назначение

Фильтр позволяет отображать бронирования в зависимости от зоны парковки.

---

## 📊 Значения зон и их смысл

| Зона | Значение | Когда используется |
| ---- | -------- | ------------------ |
| `A` | Зона А | заявки на парковку в зоне А |
| `B` | Зона В | заявки на парковку в зоне В |
| `C` | Зона С | заявки на парковку в зоне С |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Zone
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `A`

* отображаются только записи:

```json
{ "zone": "A" }
```

---

### `B`

* отображаются только:

```json
{ "zone": "B" }
```

---

### `C`

* отображаются только:

```json
{ "zone": "C" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No bookings found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Booked`, стала `Confirmed`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий бронирования в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление бронирований

Для каждой записи:

* заполнить форму
* проверить `Book Parking.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Подтверждение бронирований зоны А

* найти бронирования с `Parking Zone = A`

* для каждой найденной записи:

  * выбрать строку
  * проверить доступность `Confirm Booking`
  * изменить статус на:

```json
{ "status": "Confirmed" }
```

---

### 4. Использование фильтра

* выбрать `Filter by Zone = A`
* убедиться, что отображаются только нужные записи

---

### 5. Поиск

* выполнить поиск по имени сотрудника (например: `"Ivan"`)
* проверить корректность результата

---

### 6. Отмена одного бронирования

* выбрать любую запись со статусом `Booked`
* нажать `Cancel Booking`
* убедиться, что статус изменился на `"Cancelled"`

---

### 7. Удаление записи

* выбрать любую запись
* нажать `Delete Booking`
* убедиться, что запись исчезла

---

### 8. Проверка итогового состояния

* проверить:

  * все бронирования добавлены (10 записей)
  * бронирования зоны `A` имеют статус `"Confirmed"`
  * одна запись отменена
  * одна запись удалена
