## 🔘 Детализация кнопок управления

---

### `Resolve Issue`

**Назначение:**
Закрытие заявки после решения проблемы.

**Бизнес-смысл:**
Проблема устранена, заявка переводится в архив.

---

**Когда используется:**

* инженер устранил неполадку;
* пользователь подтвердил решение.

---

**Алгоритм работы:**

1. Пользователь выбирает строку в таблице.
2. Проверяется текущий статус заявки:

   * если `Status = Closed` → операция запрещена;
   * если `Status ≠ Closed` → операция разрешена.
3. При нажатии кнопки:

   * статус изменяется на:

```json
{ "status": "Closed" }
```

4. В поле `Processed By` записывается текущий пользователь (например, `admin`).
5. Обновляется таблица (с задержкой, если реализована асинхронность).

---

**Ограничения:**

* нельзя повторно закрыть уже обработанную заявку;
* кнопка должна быть `disabled`, если:

  * не выбрана строка;
  * статус уже `Closed`.

---

### `Assign Engineer`

**Назначение:**
Назначение ответственного инженера на заявку.

**Бизнес-смысл:**
Заявка передана конкретному специалисту для решения.

---

**Когда используется:**

* заявка прошла первичную фильтрацию;
* требуется ручное назначение исполнителя.

---

**Алгоритм работы:**

1. Выбирается строка таблицы.
2. При нажатии:

   * открывается окно выбора или автоматически назначается инженер;
   * в запись добавляется поле `Assigned To`.
3. Таблица обновляется.

---

**Ограничения:**

* кнопка `disabled`, если:

  * нет выбранной строки;
* после назначения интерфейс корректно обновляется.

---

### `Delete Issue`

**Назначение:**
Удаление заявки из системы.

---

**Бизнес-смысл:**

* удаление дубликатов;
* удаление тестовых записей;
* очистка некорректных данных.

---

**Алгоритм работы:**

1. Пользователь выбирает строку.
2. Нажимает кнопку `Delete Issue`.
3. Запись удаляется из таблицы.
4. Таблица обновляется.

---

**Ограничения:**

* кнопка `disabled`, если:

  * не выбрана строка;
* после удаления:

  * состояние интерфейса должно корректно обновляться.

## 🔍 Фильтрация заявок

Необходимо добавить блок фильтрации:

### `Filter by Priority`

**Значения:**

* `All`
* `Low`
* `Medium`
* `High`

---

### Логика

* при выборе значения отображаются только соответствующие записи
* фильтрация выполняется без перезагрузки страницы

---

## 🔍 Детализация фильтрации `Filter by Priority`

---

### 📌 Назначение

Фильтр позволяет отображать заявки в зависимости от их приоритета обработки.

---

## 📊 Значения приоритетов и их смысл

| Приоритет | Значение | Когда назначается |
| ---------- | -------------------------------- | ------------------------------ |
| `Low` | Низкий приоритет | при создании некритичной заявки |
| `Medium` | Средний приоритет | при стандартной заявке |
| `High` | Высокий приоритет | при критических проблемах |

---

## ⚙️ Алгоритм работы фильтра

---

### 1. Выбор значения

Пользователь выбирает значение в выпадающем списке:

```
Filter by Priority
```

---

### 2. Обработка выбора

#### Если выбрано:

### `All`

* отображаются все записи без фильтрации

---

### `Low`

* отображаются только записи:

```json
{ "priority": "Low" }
```

---

### `Medium`

* отображаются только:

```json
{ "priority": "Medium" }
```

---

### `High`

* отображаются только:

```json
{ "priority": "High" }
```

---

### 3. Применение фильтра

* фильтрация выполняется:

  * без перезагрузки страницы;
  * на стороне клиента (или сервера — опционально);
* обновление таблицы происходит с задержкой (имитация реального API).

---

### 4. Поведение при отсутствии данных

Если записи отсутствуют:

* отображается сообщение в текстовой метке:

```
No issues found
```

---

### 5. Взаимодействие с другими элементами

---

#### ✔ После изменения статуса

* запись может "исчезнуть" из текущего фильтра
  (например, была `Open`, стала `Closed`)

---

#### ✔ После удаления

* таблица автоматически обновляется с учётом текущего фильтра

---

#### ✔ Совместно с поиском

Если реализован поиск:

* сначала применяется фильтр по приоритету
* затем поиск по заголовку

# **Задание 2. Разработка продвинутого RPA-сценария**

Необходимо реализовать процесс автоматизации, обрабатывающий заявки в веб-приложении.

---

## 🤖 Алгоритм работы робота

### 1. Подготовка

* открыть браузер
* перейти на сайт
* дождаться загрузки страницы

---

### 2. Добавление заявок

Для каждой записи:

* заполнить форму
* проверить `Add Issue.disabled`
* при необходимости исправить данные
* отправить форму
* дождаться сообщения об успехе в текстовой метке

---

### 3. Назначение исполнителей

После добавления:

* для каждой записи в таблице:

  * выбрать строку
  * проверить доступность `Assign Engineer`
  * назначить инженера
  * убедиться, что поле `Assigned To` заполнено

---

### 4. Обработка заявок с высоким приоритетом

* найти заявки с `Priority = High`

* для каждой найденной заявки:

  * выбрать строку
  * проверить доступность `Resolve Issue`
  * изменить статус на:

```json
{ "status": "Closed" }
```

---

### 5. Использование фильтра

* выбрать `Filter by Priority = High`
* убедиться, что отображаются только нужные записи

---

### 6. Поиск

* выполнить поиск по заголовку (например: `"Login"`)
* проверить корректность результата

---

### 7. Удаление записи

* выбрать любую запись

* нажать `Delete Issue`

* убедиться, что запись исчезла

---

### 8. Проверка итогового состояния

* проверить:

  * все заявки добавлены (10 записей)
  * у всех записей заполнено `Assigned To`
  * заявки с `Priority = High` имеют статус `"Closed"`
